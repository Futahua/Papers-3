## 1. Executive decision summary

**What Papers 3 is.** Papers 3 is Papers 2 — the Electron/vanilla-JS Backpack product in `Futahua/papers-real2` — with Hermes attached as a global, Papers-native assistant layer. Nothing about Papers 2's product model changes: the world view, Backpacks, things, notes, the Desk, revisions, activity history, and the existing per-Backpack conversation all keep working exactly as they do today. Hermes is added *alongside* them as: (a) a Hermes runtime manager in the Electron main process, (b) a gateway client speaking Hermes' authenticated JSON-RPC WebSocket protocol, (c) a small Papers-owned assistant conversation store with immutable scopes, and (d) three renderer surfaces — a persistent shell-level AI button, a collapsible right-side assistant drawer, and a `Ctrl+Alt+Q` companion window that projects the same state.

**What repository becomes Papers 3.** `Futahua/papers-real2`, continued in place on a feature branch. No new repository. No history merge. (Both of those are explicitly forbidden — see §5.)

**What Papers 1 contributes.** Behavior, not code. Papers 1 proved, on the creator's real Windows machine, exactly the mechanics Papers 3 needs (each is cited in full in §3.2 and §9–§11):

* pinned, hash-verified Hermes installation into a private app-data directory (**Observed:** P1 — `src-tauri/src/runtime.rs:141–240` — `RuntimeManager::install`; P1 — `hermes.lock.json`);
* loopback-only `hermes serve` startup with a per-launch session token, `HERMES_DASHBOARD_TUI=1`, and a 30-second health loop against `GET /api/status` (**Observed:** P1 — `src-tauri/src/runtime.rs:274–361` — `RuntimeManager::start`, `runtime.rs:811–818` — `health`);
* the authenticated `ws://127.0.0.1:{port}/api/ws?token={token}` gateway URL (**Observed:** P1 — `src-tauri/src/runtime.rs:559–562` — `gateway_url`);
* a JSON-RPC 2.0 request/response + event-frame protocol with the requests `session.create`, `session.resume`, `prompt.submit`, `session.interrupt`, `approval.respond`, `clarify.respond` and the events `gateway.ready`, `message.start/delta/complete`, `tool.start/progress/generating/complete`, `approval.request`, `clarify.request`, `error` (**Observed:** P1 — `src/gateway.ts:64–144`; P1 — `src/use-agent.ts:181–354, 440–521, 558–623, 639–683`);
* private-reasoning redaction before anything is persisted (**Observed:** P1 — `src-tauri/src/storage.rs:566–606` — `sanitized_event_payload`, `redact_private_reasoning_fields`; verified by tests at `storage.rs:657–709`);
* a compact always-on-top companion window toggled by the global shortcut `Ctrl+Alt+Q` (**Observed:** P1 — `src-tauri/src/main.rs:43` — `PAPERS_GLOBAL_SHORTCUT`, `main.rs:505–524, 339–350, 650–662`; P1 — `src-tauri/tauri.conf.json` — the `companion` window entry; P1 — `src/Companion.tsx:1–78`).

**What remains unchanged.** All of P2's `app/world/*` (storage, things, revision), `app/engine/*` (the existing engine seam and its two runtimes), the entire existing IPC surface in `app/main.js:31–429`, the entire preload API in `app/preload.js:8–32`, all existing UI behavior in `app/ui/app.js`, all 34 world tests and all engine tests, and every byte of existing on-disk user data (`world.json`, `rooms/**`). The existing Backpack conversation (`room:say` → `conversation.json`) is deliberately **not** migrated in the initial attachment (§16).

**The smallest viable integration.** One new main-process subsystem (`app/hermes/`, nine small modules), one new preload namespace (`window.papersAssistant`), one new renderer file rendering into a new sibling DOM root (`app/ui/assistant.js` + `#assistant`), one companion window (`app/ui/companion.html` + `companion.js` + `app/preload-companion.js`), a pinned `app/hermes/hermes.lock.json`, appended CSS, and tests. Two lines of tracking added to existing handlers in `app/main.js`; everything else in existing files is additive registration. Two dependency changes: `yaml` (production, for safe Hermes `config.yaml` read/patch — justified in §9.7) and `ws` (dev-only, to build the fake Hermes gateway for integration tests — justified in §20).

**Largest risks** (full register in §24):

1. **Hermes protocol drift** — the lock pins Hermes 0.18.0; every protocol fact in this plan is verified against P1 code that ran against that exact version, but Papers 3 re-verifies at runtime and must fail honestly on mismatch.
2. **Streaming through P2's re-render-everything renderer** — P2's UI re-renders `#app` wholesale (`app/ui/app.js:1149–1152` — `render`); streaming deltas would be destroyed by it, which is why the assistant renders into its own root (§14).
3. **Process lifecycle on Windows** — a spawned `hermes serve` must never outlive Papers; P2 currently has no child-process lifecycle management at shutdown (its runtimes are one-shot spawns, `app/engine/runtimes/claude-cli.js:18–65`), so §9.13 defines the shutdown contract precisely.
4. **YAML config safety** — writing Hermes' `config.yaml` wrongly could break the user's Hermes; mitigated by parse-or-refuse semantics (§9.7).
5. **Scope-immutability regressions** — enforced structurally (the coordinator never consults the *active* Backpack after scope lock; §12.2) and pinned by tests.

**Recommended implementation order** (full phase detail in §21): (1) test seams + lock + paths; (2) runtime manager (install/start/stop/health); (3) gateway client + redaction + event normalization; (4) conversation store + scope model + context envelope; (5) IPC contract + preload; (6) assistant drawer with setup panel and streaming; (7) tool/approval/clarify/interrupt surfaces; (8) companion window + `Ctrl+Alt+Q`; (9) recovery/security hardening; (10) evaluate (not perform) existing-AI unification; (11) Windows packaging validation.

## 3. Evidence-based current architecture

### 3.1 Papers 2 as it exists today

**Electron main process** (`app/main.js`).

* Startup: module load resolves `worldDir` from `PAPERS_WORLD_DIR` or `path.join(app.getPath('appData'), 'Papers', 'world')` (`app/main.js:17–19`) and constructs a single `WorldStore` (`app/main.js:21`). `app.whenReady()` (`app/main.js:484–501`) either runs the headless smoke check (`--papers-smoke`, `app/main.js:485–494`, exercising `smokeCheck` at `433–445`) or loads the world (`world = store.loadWorld()`, line 495), creates the one window (`createWindow`, `app/main.js:447–468` — 1240×840, `backgroundColor '#f5f0e6'`, menu bar hidden, `preload.js`, `contextIsolation: true`, `nodeIntegration: false`), and registers all IPC handlers (`registerHandlers(() => win)`, line 497). `app.on('activate')` re-creates the window on macOS-style re-activation (`app/main.js:498–500`).
* Single instance: `app.requestSingleInstanceLock()` at top level; the second instance logs and `app.exit(1)` before any world write (`app/main.js:479–482`, with the rationale comment at 470–478).
* Shutdown: `window-all-closed` → `app.quit()` (`app/main.js:503–505`). **Observed:** there are no `before-quit`/`will-quit` handlers and no long-lived child processes today — every AI call is a one-shot spawn or fetch that resolves or times out on its own.
* IPC surface (all `ipcMain.handle`, request/response only, no main→renderer push events anywhere today): `world:get`, `world:createRoom`, `room:get` (marks entered), `room:refresh`, `room:rename`, `room:setDescription`, `room:pinNote`, `room:attachThings` (native `dialog.showOpenDialog`, `app/main.js:62–73`), `room:detachThing`, `room:openThing` (`shell.openPath`/`showItemInFolder` with honest missing-reality refusal, `82–99`), `room:previewThing`, `desk:add`, `desk:remove`, `desk:setBrief`, `desk:synthesizePreview`/`desk:synthesize` (guarded two-step, `156–236`), `note:revisePreview`/`note:revise` (guarded two-step bound to a sha256 prompt fingerprint, `251–314`), `room:noteSources`, `world:openRealPath`, `room:say` (`354–367`), `room:notePreview`/`room:createNote` (guarded two-step, `372–428`).

**Preload boundary** (`app/preload.js`).

* One `contextBridge.exposeInMainWorld('papers', {...})` object of 22 functions, each a 1:1 `ipcRenderer.invoke` mapping to the handlers above (`app/preload.js:8–32`). No Node APIs, no events, no raw `ipcRenderer` exposure. This is the pattern Papers 3 extends (§10).

**Renderer** (`app/ui/app.js`, `index.html`, `style.css`).

* One mutable `state` object (`app/ui/app.js:12–30`) with `view: 'world' | 'room'`, the loaded room view, selection state, guard state, `talkOpen` persisted in `localStorage` (`papers-ui:talk-open`, line 29).
* Rendering is wholesale: `render()` (`app/ui/app.js:1149–1152`) calls `renderWorld()` (`99–143`) or `renderRoom()` (`699–776`), each of which sets `appEl.innerHTML` for the entire `#app` div and re-wires listeners. **Risk (for streaming):** any state under `#app` is destroyed on every render; this drives the §14 decision to render the assistant into a sibling root.
* The Backpack (room) surface is primary: overview (`roomOverviewHtml`, `301–365`), Desk (`deskSectionHtml`, `416–462`), things (`thingRow`, `147–166`), notes (`noteCard`, `178–197`), history timeline (`roomTimelineHtml`, `253–281`), note surface (`noteSurfaceHtml`, `527–547`), preview surface (`previewSurfaceHtml`, `597–644`). The conversation is a right-side, fixed-width (340px), tuckable panel (`talkPanelHtml`, `672–695`; CSS `.room-talk`, `app/ui/style.css:153–170` with the comment "It must never grow into the shell").
* Guarded actions render a modal (`renderGuard`, `1025–1082`; `wireGuard`, `1084–1140`).
* `index.html` is 13 lines: strict CSP, one `#app` div, one script tag (`app/ui/index.html:1–13`).

**World storage** (`app/world/store.js`).

* System of record on disk, human-legible JSON: `world.json`; per room `rooms/<roomId>/room.json`, `things.json`, `conversation.json`, `activity.json`, `desk.json`, `artifacts/<id>.json` (layout comment `app/world/store.js:14–23`).

* `loadWorld` creates the world on first open and stamps `lastOpenedAt` (`61–70`). `listRooms` aggregates room cards (`72–96`). `createRoom` seeds the four room files (`98–113`). `markRoomEntered` shifts `previousEnteredAt`/`lastEnteredAt` (`134–140`).

* `getRoomView` assembles room+things(refreshed)+artifacts+conversation+desk+activity (`537–547`).

**Conversation persistence & existing AI flow** (the flow Hermes must *not* disturb).

1. Renderer submit (`wireTalk` form handler, `app/ui/app.js:1000–1020`) optimistically echoes the creator entry, sets `state.thinking`, calls `window.papers.say(roomId, text)`.
2. Main `room:say` (`app/main.js:354–367`) persists the creator entry, calls `engine.roomReply(roomContext(roomId), text)`, persists either the `ai` entry (with `meta.engine` provenance) or a `status` entry with the honest error, returns the whole persisted conversation.
3. Renderer replaces its conversation array with the persisted record and re-renders. **Observed:** the round trip is fully blocking per message; there is no streaming anywhere in P2 today.

**Engine abstraction** (`app/engine/index.js`, `prompts.js`).

* The "engine seam" is the only door to any AI runtime (file comment `app/engine/index.js:3–17`). Runtime registry is built from `claude-cli` and `ollama` (`21–24`), selected by `PAPERS_ENGINE` env with default `claude-cli` (`26–30`). The runtime contract is `{ id, label(), complete(prompt) → {ok, text, model?} | {ok:false, error} }`. `complete()` refuses unknown runtimes by name, honestly (`32–47`). Papers-native actions: `roomReply` (`52–54`), `generateRoomNote` (`57–62`), `synthesizeDesk` (`66–71`), `reviseNotePrepared` (`82–87`, which takes an already-approved exact prompt string).
* `prompts.js` serializes room context truthfully (`roomContextBlock`, `12–68`, including the last 12 non-status conversation entries at `56–63`) and builds/parses each action prompt (`replyPrompt` `70–77`, `notePrompt` `82–102`, `deskSynthesisPrompt` `120–159`, `reviseNotePrompt` `173–206` — deliberately context-free, `parseNoteResponse` `106–115`).

**Runtime adapters.**

* `claude-cli.js`: spawns the locally installed Claude Code CLI in print mode; Windows path via `cmd.exe /c claude -p --output-format text`; prompt over stdin only; 240 s timeout; honest sign-in and failure messages (`app/engine/runtimes/claude-cli.js:10, 18–65, 67–88`).

**Revision system** (`app/world/revision.js`).

* `gatherRevisionMaterial` re-reads the note's recorded sources from reality, builds the exact runtime prompt through `reviseNotePrompt`, and fingerprints the exact prompt bytes with sha256 (`37–69`); `approvalMatches` requires the fingerprint to match (`73–75`); the constants `REVISION_CHANGED_ERROR` etc. live at `24–29`. **UI state & shutdown.** Covered above; there is no renderer-side persistence except `localStorage['papers-ui:talk-open']` (`app/ui/app.js:29, 986`).

**Tests** (all `node:test`).

* `app/test/engine.test.js` (lines 1–227): truthful context serialization, Desk weighting, revision prompt self-containment (`134–150`), honest failure propagation for unknown/unreachable runtimes, TITLE parsing.
* `app/scripts/verify-slice1.js`, `verify-desk.js`, `verify-note-revision.js`: headless end-to-end verifications that boot Electron with a temp world.
* **Observed:** there are no tests for `main.js` IPC handlers themselves and no UI tests; the store and engine seams are the tested seams. Papers 3 follows the same pattern (§20).

### 3.2 Papers 1's Hermes integration as it exists today

**Install & version locking.** `hermes.lock.json` (16 lines, quoted in §2.2) is compiled into the binary via `include_str!` (`src-tauri/src/runtime.rs:37–38`). `RuntimeManager::install` (`runtime.rs:141–240`): short-circuits if `hermes.exe` already exists (writes guarded config, ensures Computer Use, returns status, `142–146`); preserves any partial install by renaming it into `staging/hermes-agent-partial-<ts>` (`preserve_partial_install`, `242–257`); writes an isolated `installer.gitconfig` (`core.autocrlf=false`, `windows.appendAtomically=false`) used via `GIT_CONFIG_GLOBAL` so the user's Git settings are untouched (`259–272`, env at `202`); downloads the pinned installer, verifies SHA-256 against the lock and refuses on mismatch with "Nothing was run." (`152–170`); runs `powershell.exe -NoLogo -NoProfile -ExecutionPolicy Bypass -File <installer> -Commit <lock.commit> -HermesHome <paths.hermes_home> -InstallDir <paths.hermes_install> -SkipSetup -NonInteractive` with `HERMES_HOME` set, stdout/stderr appended to `logs/hermes-install.log` (`177–228`); filters `WindowsApps`, `chocolatey\bin`, and `scoop\shims` out of the installer's `PATH` so package-manager shims can't be used (`206–208`, `940–953`, unit-tested at `971–994`); snapshots and restores the user's `HERMES_HOME` and `Path` user-environment values around the install (`178–216`, `955–969`, `996–1021`); verifies the executable exists afterwards (`230–235`).

**Runtime paths.** `PapersPaths::discover` (`src-tauri/src/paths.rs:28–52`): root = `PAPERS_DATA_HOME` or `%LOCALAPPDATA%\Papers`; `data/`, `data/hermes` (HERMES_HOME), `runtime/`, `runtime/hermes-agent` (install dir), `staging/`, `versions/`, `logs/`; `hermes_executable()` = `<install>/venv/Scripts/hermes.exe` (`70–75`).

**Process startup, port, token.** `RuntimeManager::start` (`runtime.rs:274–361`): requires installed executable; reuses a live child (`280–301`); writes the guarded config (§9.7); picks an ephemeral port by binding `127.0.0.1:0` and reading the assigned port (`available_port`, `844–849`); generates the session token as two concatenated UUIDv4s (`305`); spawns `hermes.exe serve --host 127.0.0.1 --port <p>` with `current_dir = <root>/runtime` (deliberately *not* a repo checkout — comment history in P1 `PROJECT_CONTEXT.md`), env `HERMES_HOME`, `HERMES_EXEC_ASK=1`, `HERMES_DASHBOARD_TUI=1` (required or Hermes closes the WebSocket immediately — **Observed:** P1 `PROJECT_CONTEXT.md`, "What the current app honestly does" bullet on `hermes serve`), `HERMES_DASHBOARD_SESSION_TOKEN=<token>`, `PLAYWRIGHT_BROWSERS_PATH=<root>/runtime/playwright`, `HERMES_CUA_DRIVER_CMD=<cua path>`; logs appended to `logs/hermes-serve.log`; console hidden on Windows (`CREATE_NO_WINDOW`, `935–938`).

**Health verification.** Startup polls `GET http://127.0.0.1:{port}/api/status` every 250 ms up to 120 times (30 s), and simultaneously watches for child exit, failing with the log path in the message either way (`runtime.rs:338–360`; `health` at `811–818`). `status()` (`74–139`) derives the phase string: `not_installed` / `stopped` / `starting` (running but not healthy) / `ready` (running and healthy), plus `installing` and `error` used by the frontend type (`src/types.ts:12–18`).

**Gateway connection.** `gateway_url()` = `ws://127.0.0.1:{port}/api/ws?token={token}` (`runtime.rs:559–574`). The native `GatewayProxy` (`src-tauri/src/gateway_proxy.rs:30–97`) connects with a 15 s timeout, splits the socket, pumps outgoing frames from an unbounded channel, answers WebSocket pings with pongs (`64–69`), emits every text/binary frame to the webview as the Tauri event `papers://gateway-frame`, and emits `papers://gateway-closed` with an honest reason on any close/error (`87`). `send` fails with "Hermes is not connected" when there is no live sender (`99–111`); `disconnect` drops the sender and aborts the task (`113–123`). The token never reaches the webview.

**JSON-RPC construction & correlation.** Frontend `HermesGateway.request(method, params, timeoutMs=120000)` (`src/gateway.ts:64–96`): per-connection incrementing integer `id`; frame `{jsonrpc:"2.0", id, method, params}`; a `pending` map resolves/rejects on the response frame (`handleMessage`, `116–137`); unmatched response ids are ignored (`126–128`); malformed JSON is silently dropped (`118–121`); on close, all pending calls reject (`rejectPending`, `146–152`). Event frames are `{method:"event", params:{type, session_id?, payload?}}` and fan out to handlers (`139–144`; the `GatewayEvent`/`GatewayFrame` types at `src/types.ts:287–300`).

**Session creation / resumption / prompts / interrupt / approvals / clarifications.** All in `src/use-agent.ts`:

* `session.create` params `{cols: 96, source: "papers"|"papers-builder", title, cwd?, profile?}`, 60 s timeout; response `{session_id, stored_session_id?, messages?}` (`SessionCreateResponse`, `22–31`; call at `469–496`). The returned `stored_session_id ?? session_id` is bound to the local session row (`bind_hermes_session`, `484`).
* `prompt.submit` params `{session_id, text}`, 1,800 s timeout (`514–518`, and `588–592`). Optional context is prepended as plain text: `` `${options.context}\n\n${text}` `` (`498`).
* `session.interrupt` params `{session_id}` — used by both Stop (`558–567`) and Pause (`569–578`, which also queues a canned continue prompt for Resume, `580–593`).
* `approval.respond` params `{choice: "once"|"deny", session_id}` (`595–608`).
* `clarify.respond` params `{request_id, answer}` (`610–623`).
* `session.resume` params `{session_id, cols: 96}`, 60 s timeout; response includes `session_id` (possibly new) and `messages[]` with `{role, text|content, timestamp}` which are filtered to user/assistant and replayed into the UI (`639–683`).

**Streaming events → UI state.** `handleEvent` (`181–354`): drops events whose `session_id` doesn't match the active Hermes session (`185–187`); records every event to SQLite through the redaction gate (`189–192`); `gateway.ready` → activity note; `message.start` → run state `planning`, create pending assistant message; `message.delta` → append `payload.text ?? payload.rendered ?? payload.message` (via `payloadText`, `44–47`) to the pending message, ignored when no pending message exists (`218–220`); `message.complete` → finalize text (prefer final text), run state `completed`; `tool.start|progress|generating` → run state `acting`, tool card "running" with humanized name (`readableToolName`, `54–59`); `tool.complete` → tool card done + optional artifact detail (`artifactDetail`, `85–94`); `approval.request` → run state `awaiting_approval`, approval card from `payload.command`/`description`/`allow_permanent`; `clarify.request` → clarify card from `payload.request_id`/`question`/`choices[]`; `error` → run state `failed` with honest message; unknown types → ignored (`349–351`).

**Persistence & redaction.** SQLite `papers.db` (`src-tauri/src/storage.rs:15–99`): `sessions` (id, hermes_session_id, title, mode, state, timestamps), `events` (per-session monotonically numbered, `263–293`), `approvals`, `changes`, `sync_queue`, `pending_oauth`, `provider_tests`. Run state transitions are whitelist-validated (`237–261`). **Redaction:** `record_event` refuses to store whole event types with prefixes `reasoning.`, `thinking.`, `chain_of_thought.` and recursively strips the keys `reasoning`, `thinking`, `chain_of_thought`, `chainOfThought`, `cot` from every stored payload (`566–606`); tests prove both (`657–709`). The decision log records the original violation and the scrub (P1 `PROJECT_CONTEXT.md`, 2026-07-07 entry).

**Companion window.** Declared in `src-tauri/tauri.conf.json`: label `companion`, `index.html?companion`, 430×112 fixed, undecorated, transparent, always-on-top, skip-taskbar, initially hidden. `Ctrl+Alt+Q` registered as a global shortcut (`main.rs:43, 505–531`); the handler remembers the foreground window title (for "Ask Papers about this app…") and toggles visibility (`toggle_companion`, `339–350`), positioning bottom-right with monitor-scale-aware margins (`position_companion`, `650–662`). The Companion React component (`src/Companion.tsx:1–78`) shares state with the main window over a `BroadcastChannel("papers-agent")`: main window broadcasts `{type:"status", status, state, active}` on every run-state change (`src/App.tsx:161–168`) and handles `pause/resume/stop/new/quickPrompt` messages from the companion (`170–183`); the companion posts `quickPrompt` with the remembered foreground target, shows status, and offers Pause/Stop/Expand(`show_main`)/Hide buttons.

**Shutdown.** On Tauri `RunEvent::Exit`, the app disconnects the gateway and kills the Hermes child (`main.rs:605–610`; `RuntimeManager::stop` at `runtime.rs:363–375` kills and reaps the child and clears port/token).

**Provider/model configuration (context for §9.7 and §6).** Hermes' own files under HERMES_HOME are the source of truth: `config.yaml` (model/provider under a `model:` map — `model_config_values`, `runtime.rs:851–874`; defaults `nous` / `stepfun/step-3.7-flash:free`, `runtime.rs:18–19`) and `auth.json` (`active_provider`, `providers` map — read-only sanitized reads at `600–632`). Papers 1 writes `config.yaml` only through guarded, atomic writes (`write_main_config`, `588–598`; `write_guarded_config`, `738–809`, which pins `approvals: {mode: manual}`, toolsets, and `agent.yolo: false`). Nous sign-in is a device-code OAuth loop through Hermes' loopback HTTP API (`start_nous_login`, `377–459`: POST `/api/providers/oauth/nous/start` with bearer token, open browser, poll `/api/providers/oauth/nous/poll/{session_id}` until approved/denied/expired). The larger provider-orchestration layer (`provider_catalog.rs`, `provider_service.rs`, `provider_state.rs`, `hermes_provider_adapter.rs`, `SettingsWizard.tsx`) exists and works against Hermes HTTP endpoints (`/api/providers/oauth`, `/api/model/options`, `/api/model/set`, `/api/providers/validate`, `/api/env` — `hermes_provider_adapter.rs:194–553`) but is **explicitly not ported** in the initial Papers 3 scope (§5, §8.5): it is a provider dashboard, which the product objective forbids as a primary experience; Papers 3 ports only the minimal first-run path (install → start → Nous sign-in → model status line → one live echo test).

**What Papers 1 is NOT (and what must not be copied).** The three-column agent workbench (`src/App.tsx` — session rail / conversation / Work rail), the Inspect/self-edit system (`self_edit.rs`, `inspect.ts`, `policy.rs`, `mcp.rs`, builder profile), the provider wizard, the recovery `launcher/` crate, and the SQLite storage model are Papers 1's *application identity*, which P2's governing docs reject ("a generic agent workbench with Papers vocabulary painted on top" — P2 `README.md`). They are listed as not-ported in §8.5.

## 4. Product invariants

Every item below must remain true at every phase boundary in §21. Each is **Decision** unless marked otherwise.

1. **Backpacks remain primary.** The Backpack surface stays the main working surface; the assistant drawer is a collapsible overlay region and the companion is a small projection. Neither may become the shell. (Mirrors P2's existing conversation rule — **Observed:** CSS comment "It must never grow into the shell", `app/ui/style.css:151–152`.)
2. **One active Backpack at a time**, tracked and switched by Papers exactly as today (`state.view`/`state.roomId`, `app/ui/app.js:12–30`; `room:get` marks entry, `app/main.js:40–43`). Papers 3 adds main-process *observation* of this fact (§7.4) and changes nothing about how it works.
3. **Existing data remains readable.** Every file P2 writes today (`world.json`, `rooms/<id>/room.json|things.json|conversation.json|activity.json|desk.json|artifacts/*.json`) keeps its schema, location, and writer code untouched. New assistant state lives only in new files under `<worldDir>/assistant/` (§17).
4. **Existing Backpack workflows remain usable with Hermes absent, stopped, or broken.** Attach/preview/note/Desk/revise/say flows never call any `app/hermes/*` module. (Enforced by test — §20.1, mirroring the require-cache technique of `app/test/world.test.js:585–594`.)
5. **Hermes is global.** The AI button, drawer, and `Ctrl+Alt+Q` companion are reachable from the world view and from inside any Backpack.
6. **Conversation scope is immutable after the first submitted message** — `global` or `backpack:<roomId>`, chosen at creation, locked at first successful submission (§12.2).
7. **Switching the active Backpack never retargets or cancels an existing conversation or run.** The coordinator resolves context from the conversation's own stored scope, never from the currently active Backpack, after lock (§12.5). Runs continue across Backpack switches.
8. **Context is explicit and bounded.** Only the envelope defined in §13 is sent: scope identity, capped Backpack metadata, explicitly attached resources (read through the existing truthful `readThingContent` — `app/world/things.js:58–118`), and the instruction. No conversation dumps, no world dumps, no filesystem walks.
9. **Hermes owns agent execution** — tools, terminals, computer use, delegation to Codex/Claude Code/OpenCode, model/provider execution, approvals semantics, session memory. Papers renders and responds; it never re-implements a tool loop.
10. **Papers does not recreate coding agents.** No terminal panel, no patch protocol, no process manager, no provider dashboard in Papers 3. (Also a standing rule of the P2 repo — P2 `README.md` hard rule 8.)
11. **No hidden reasoning is persisted or displayed.** Every event crossing into persistence or renderer passes the redaction gate ported from P1 (`storage.rs:566–606`) — §7.9, §18.
12. **Failures are visible and honest.** Every Hermes failure surfaces as a labeled, dismissible state with a recovery action, in the same voice as P2's existing honest errors ("Papers will not invent a reply in its place", `app/engine/runtimes/claude-cli.js:67–73`). No silent retries that fake liveness.
13. **Installation data, credentials, and runtime state are never committed.** The Hermes install tree, HERMES_HOME (contains `auth.json`), logs, tokens, and ports exist only under the local app-data root (§9) and are covered by `.gitignore` policy (P2 `.gitignore` already excludes `auth.json`, `.env*`, `*.key`); the only committed Hermes artifact is `app/hermes/hermes.lock.json` (hashes and URLs, no secrets — same footprint as P1's committed `hermes.lock.json`).
14. **The renderer never sees the gateway token, the Hermes port, raw sockets, or child-process handles.** Same boundary as P1 (**Observed:** token lives only in Rust, `gateway_proxy.rs`; P1 `PROJECT_CONTEXT.md` "never receives the gateway token"). In Papers 3 those live only in the Electron main process (§10, §18).
15. **The world store remains the only custodian of Papers continuity** (engine seam comment, `app/engine/index.js:10–13`): losing or reinstalling Hermes must never lose Papers data. Hermes session transcripts remain Hermes-owned; Papers stores only its own projection (§12.8).

## 5. Scope and non-scope

### 5.1 Required for Papers 3 (initial attachment)

1. Pinned Hermes install/verify/start/stop/health lifecycle in the Electron main process (§9), private to Papers' app-data directory.
2. Gateway client (JSON-RPC over loopback WebSocket, token-authenticated) as an explicit state machine (§11).
3. Event normalization + reasoning redaction (§7.8–7.9).
4. Papers-owned assistant conversation store with immutable scope, resumable Hermes session binding, and a local redacted transcript projection (§12, §17).
5. Explicit context envelope builder (§13).
6. Narrow preload/IPC contract (`window.papersAssistant`) with main→renderer event push (§10).
7. Shell-level AI button + right-side assistant drawer: setup panel, conversation list scoped Global/this-Backpack, streaming text, tool activity lines, approval card, clarification card, interrupt, resource attachment (§14).
8. Companion window on `Ctrl+Alt+Q`, projecting the same coordinator state (§15).
9. First-run setup path: install → start → Nous sign-in → provider/model status → one live echo test (§6.2–6.4, §9).
10. Failure/recovery behavior per the matrix (§19) and security boundaries (§18).
11. Tests at all four levels (§20) and Windows packaging validation (§21 Phase 11).

### 5.2 Deferred (valuable, but not in the initial attachment)

* **Computer Use (Cua driver) install and the `computer_use` toolset.** P1 installs a pinned cua-driver and passes `HERMES_CUA_DRIVER_CMD` (`runtime.rs:634–736, 324`). Papers 3 v1 configures toolset `["hermes-cli"]` only; the lock file keeps the `computerUse` block so a later phase can port `install_computer_use` unchanged. Rationale: smallest reliable attachment; computer use adds an installer, an archive verify, and a large behavioral surface that is not needed to prove the attachment.
* **Guided setup for providers beyond Nous** (the P1 catalog/wizard). Papers 3 shows honest read-only provider/model status and offers Nous sign-in; other providers get the honest line P1 uses: configure through Hermes for now (`runtime.rs:534–542`).
* **Model picking UI via Hermes `/api/model/options` + `/api/model/set`** (endpoints proven in `hermes_provider_adapter.rs:218–276, 461–468`). v1 reads current provider/model from `config.yaml` and can set them via one guarded write (§9.7); a picker UI is follow-up.
* **Routing the existing Backpack engine actions (room reply, notes, Desk synthesis, revision) through Hermes.** Evaluated in §16; explicitly not performed until the attachment has parity evidence.
* **Foreground-app targeting for the companion** (P1 `foreground_app`, `main.rs:363–377, 664–690`) — needs a native module or PowerShell probe in Electron; the v1 companion targets conversations, not foreign windows.
* **Hermes runtime upgrades** beyond the pinned version (lock policy is `manual-after-compatibility-check`); an upgrade flow is follow-up work.
* **Multi-window drawer state sync beyond the one main window + companion** (P2 can only have one main window today — `createWindow`, `app/main.js:447–468`).
* **Auto-start with Windows, tray icon** (also absent from P1 — P1 `PROJECT_CONTEXT.md` "What the current app does not do").

### 5.3 Explicitly forbidden

* Replacing Electron with Tauri, or vanilla JS with React; any bundler introduction.
* Importing P1's three-column workbench, session rail, Work rail, Inspect/self-edit system, builder MCP, policy engine, recovery launcher, or provider wizard as product surfaces.
* Merging the repositories, creating a third repository, or converting P2 into P1's architecture.
* Rewriting the Backpack storage system or migrating `conversation.json` in the initial attachment (§16, §17).
* A terminal emulator, process manager, patch/diff protocol, or coding-agent framework inside Papers.
* Exposing raw Hermes infrastructure (ports, tokens, config internals, provider plumbing) as primary UI.
* Replacing the existing claude-cli/ollama runtimes "for purity" — they stay (§16).
* Any migration that rewrites existing user data in place (§17.6).

## 6. Target user experience

All copy below is concrete enough to implement; strings may be polished during implementation but state transitions and affordances are binding. The assistant surfaces use P2's existing visual language (paper palette, serif, quiet buttons — `app/ui/style.css:3–43`).

**6.1 First launch before Hermes is installed.** Papers opens exactly as today into the world view (`renderWorld`). A new quiet button sits at the top-right edge of the shell: **"Assistant"** with a small status dot (gray = not installed). Clicking it (or pressing `Ctrl+Alt+Q`) opens the right drawer showing the setup panel: "**Papers can attach its own private AI runtime (Hermes 0.18.0).** It installs into Papers' own folder, never touches your PATH, and only ever listens on this machine (127.0.0.1)." Buttons: **[Install Hermes]** · [Not now]. Nothing else in Papers changes; every Backpack feature works as before.

**6.2 Installing the private Hermes runtime.** Clicking Install shows phase `installing` with a spinner line "Downloading the pinned installer and verifying its fingerprint…" then "Installing into Papers' private runtime… (this can take a few minutes)". Failure states are honest and terminal per attempt: checksum mismatch shows "The installer did not match its pinned fingerprint. Nothing was run." with **[Try again]**; installer exit failure shows the log path (`…\logs\hermes-install.log`) and **[Open log folder]** + **[Try again]** (behavior ported from `runtime.rs:164–235`). Success advances to 6.3 automatically.

**6.3 Connecting / configuring a model.** The setup panel shows Hermes status "Installed · starting…" then "Running". It then reads sanitized provider status (ported from `runtime.rs:461–499`): if no verified sign-in, it shows "**Sign in to Nous** to give Hermes a model. A browser window will open; Papers never sees the credential." **[Sign in with Nous]** runs the ported device-code loop (`runtime.rs:377–459`): opens the browser, shows "Waiting for the browser sign-in… code: XXXX-XXXX", and resolves to approved/denied/expired with honest messages. Provider/model line shows e.g. "nous · stepfun/step-3.7-flash:free" (defaults from `runtime.rs:18–19`). Other providers: one line — "Other providers can be configured in Hermes itself for now."

**6.4 Verification.** A final setup row: **[Run a live test]** sends a fixed echo prompt ("Reply with exactly `PAPERS_ASSISTANT_TEST_OK`, then stop." — pattern from P1 `SettingsWizard.tsx:25–27`) through a throwaway Hermes session and reports "Tested working · <timestamp>" or the honest failure. The result is persisted in setup state (§17.3) so the drawer's status dot can be truthful across restarts without re-testing.

**6.5 Opening the global assistant drawer.** The Assistant button toggles a right-side drawer (fixed overlay, 380px, full height, collapsible with the same visual grammar as the room conversation panel). Header: status dot + word ("Ready" / "Starting" / "Offline" / "Needs setup"), a conversation selector, and a [New conversation] button. The drawer is available in the world view and inside any Backpack. Its open/closed state persists in `localStorage['papers-ui:assistant-open']` (same idiom as `talkOpen`, `app/ui/app.js:29`).

**6.6 Beginning a global conversation.** In the world view, [New conversation] creates a conversation with scope **Global** (shown as a chip: "Global — this whole world"). The composer placeholder: "Ask the assistant anything…". The scope chip includes "(locks with your first message)" until locked.

**6.7 Beginning a Backpack-scoped conversation.** Inside a Backpack, [New conversation] defaults the scope selector to **This Backpack: "<title>"**, with an explicit toggle to Global before anything is sent. After the first successful submission the chip loses its toggle and reads e.g. "Backpack: Field notes" permanently. A user in Backpack A may open the selector and start/switch to a conversation scoped to Backpack B or Global at any time — the drawer lists conversations grouped: **This Backpack** / **Global** / **Other Backpacks** (§12.4).

**6.8 Attaching a resource.** A paper-clip button in the composer opens the resource picker: for a Backpack-scoped conversation it lists that Backpack's things (name, type, present/missing dot — same rows as `thingRow`, `app/ui/app.js:147–166`); for Global scope it opens the native file dialog (same pattern as `room:attachThings`, `app/main.js:62–73`). Each attached resource renders as a chip with the exact honest share detail — "plan.txt · 3,412 of 3,412 bytes will be shared" / "photos/ · listing of 200 of 412 entries" / "binary — name and size only" — computed by the same `readThingContent` truth rules (`app/world/things.js:58–118`) and the same wording as the existing guard previews (`app/main.js:165–177`). Chips have an ✕ to detach before sending. Resources are attached per message, not silently inherited forever (§13.4).

**6.9 Receiving streamed text.** On send, the user's message appears immediately; an assistant bubble appears with a typing indicator on `message.start` and grows with each `message.delta`; `message.complete` finalizes it (preferring the event's final text — behavior from `use-agent.ts:230–244`). The composer stays enabled for interrupt but blocks new submissions while a run is active in *this* conversation.

**6.10 Observing tool activity.** Between message events, tool events render as compact activity lines under the current assistant bubble: "● Command — running", "✓ Command finished", with the humanized tool name and optional one-line detail (port of `readableToolName`/`toolDetail`, `use-agent.ts:54–84`). No terminal, no raw output panel; a "details" disclosure shows the redacted one-line detail only.

**6.11 Responding to approval.** An `approval.request` renders an inline approval card pinned above the composer: eyebrow "Hermes is asking to act", the description, the exact `command` in a `<pre>` block when present, and two buttons: **[Don't allow]** / **[Allow once]** → `approval.respond {choice:'deny'|'once', session_id}` (`use-agent.ts:595–608`). The conversation status chip reads "Waiting for you". No "always allow" in v1 even when `allow_permanent` is true — smallest surface, one honest choice pair.

**6.12 Responding to clarification.** A `clarify.request` renders a clarify card: the question, choice buttons when `choices[]` present, and a free-text input; submit → `clarify.respond {request_id, answer}` (`use-agent.ts:610–623`).

**6.13 Interrupting a run.** While a run is active, the composer shows a **[Stop]** button → `session.interrupt {session_id}` (`use-agent.ts:558–567`). The run ends with a visible status line "You stopped this run." and the conversation returns to idle. (Pause/Resume as in P1 `use-agent.ts:569–593` is **Deferred** — Stop covers the safety need; Pause is a canned-prompt trick, not a protocol primitive.)

**6.14 Switching Backpacks during a run.** The user leaves Backpack A (drawer conversation scoped to A, run streaming) and enters Backpack B. The drawer does not change conversations by itself; a subtle line under the header notes "This conversation is scoped to *Field notes* — you're now in *Ledger*." The run keeps streaming. The conversation list regroups so B's conversations are under "This Backpack". Nothing is cancelled, nothing retargets (§12.5, invariant 7).

**6.15 Opening the companion.** `Ctrl+Alt+Q` toggles a small always-on-top window (≈460×180, bottom-right, frameless) whether or not the main window is focused. It shows: status dot + status word, the active conversation's title + scope chip, the last line of streamed text (or the pending approval/clarify notice), buttons [Stop] [Open Papers] [Hide], and a one-line prompt input that submits to the *same* active conversation. If a pending approval or clarification exists, the companion shows "Waiting for you in Papers — open to review" plus [Open Papers] (approvals are answered only in the full drawer in v1 — keeps the approval surface single and exact).

**6.16 Resuming a conversation.** Selecting an older conversation loads the local projection instantly (works offline), and — when Hermes is `ready` and the conversation has a bound Hermes session — issues `session.resume {session_id, cols:96}` and reconciles (behavior from `use-agent.ts:639–683`): the drawer shows "Resumed — Hermes remembers this conversation" or, on failure, "Hermes no longer remembers this conversation. You can read it here; a new message will start a fresh Hermes session with the same scope." (§12.7).

**6.17 Hermes offline (installed but not running / crashed between runs).** Drawer header shows "Offline" with **[Start Hermes]**. Conversations remain readable; composer disabled with the reason inline.

**6.18 Hermes crashing mid-run.** The gateway emits closed; the coordinator marks the active run `interrupted` with status line "Hermes stopped unexpectedly. The run was interrupted." and offers **[Start Hermes]**, then **[Try again]** (§11, §19 rows 9–10). Auto-restart is attempted at most once per crash, never in a loop.

**6.19 Model/provider failure.** An `error` event (or a prompt.submit rejection) renders as an honest status line inside the conversation ("Hermes reported: <message>") plus, when the sanitized provider status indicates a sign-in problem, a hint "Check the assistant setup panel." No invented text, ever (invariant 12).

**6.20 Delegated Codex / Claude Code / OpenCode work.** These arrive as ordinary tool events from Hermes (Hermes owns delegation — product objective). Papers renders them exactly like other tools ("Claude Code — running…"), with no special casing beyond the humanized name. Papers must not attempt to manage, install, or configure those tools.

**6.21 Closing and reopening Papers.** On quit, Papers interrupts active runs best-effort, disconnects the gateway, and kills the Hermes child (§9.13, port of `main.rs:605–610`). On reopen: world view as today; drawer restores its open/closed state; the last active conversation is preselected; status recovers via the runtime status probe; Hermes auto-starts only if it was installed AND the user enabled "Start Hermes with Papers" in the setup panel (default ON after first successful setup — persisted in §17.3; P1 auto-starts unconditionally when installed, `use-agent.ts:380–397`; the toggle is Papers 3's one refinement so the assistant never becomes an unremovable cost).

## 7. Proposed architecture

The smallest architecture that connects Electron P2 to Hermes, preserving invariant 14 (secrets in main only):

```
┌──────────────────────────────────────────────────────────────────────────────┐
│ Electron MAIN process (app/)                                                 │
│                                                                              │
│  main.js (existing)                                                          │
│   ├── WorldStore (existing, app/world/store.js) ──────────── world JSON      │
│   ├── engine (existing, app/engine/*)  ← untouched                           │
│   └── NEW wiring only:                                                       │
│        createAssistant({app, store, worldDir}) + shutdown hooks              │
│                                                                              │
│  app/hermes/ (NEW subsystem — "the assistant")                               │
│   ├── paths.js        Hermes dirs under %LOCALAPPDATA%\Papers                │
│   ├── hermes.lock.json  pinned versions/URLs/hashes (from P1)               │
│   ├── runtime.js      install / start / stop / health  ── spawns ──┐         │
│   ├── gateway.js      WS JSON-RPC client (token stays here)        │         │
│   ├── redact.js       reasoning redaction (port of storage.rs)     │         │
│   ├── events.js       gateway frame → normalized assistant event   │         │
│   ├── conversations.js  Papers-owned store: scope, session bind,   │         │
│   │                     redacted transcript projection             │         │
│   ├── context.js      explicit context envelope builder            │         │
│   ├── coordinator.js  ties all of the above; run lifecycle;        │         │
│   │                     active-backpack observation; event fanout  │         │
│   └── ipc.js          ipcMain.handle('assistant:*') + push events  │         │
│                                                                    │         │
│  preload.js (existing + NEW papersAssistant namespace)             │         │
│  preload-companion.js (NEW, assistant subset only)                 │         │
└────────────┬───────────────────────────────┬───────────────────────┼─────────┘
             │ contextBridge                 │ contextBridge         │ spawn
┌────────────▼───────────┐   ┌───────────────▼──────────┐   ┌────────▼────────┐
│ RENDERER: main window  │   │ RENDERER: companion       │   │ hermes.exe serve │
│  #app   ← app.js       │   │  companion.html/js        │   │ 127.0.0.1:<port> │
│  (existing, untouched  │   │  (projection of the same  │   │ token-authed WS  │
│   render cycle)        │   │   coordinator state)      │   │ /api/ws          │
│  #assistant ← NEW      │   └───────────────────────────┘   │ /api/status      │
│   assistant.js drawer  │                                   │ /api/providers/… │
└────────────────────────┘                                   └──────────────────┘
```

Data flow for one prompt: drawer composer → `papersAssistant.send(conversationId, text, resourceIds)` → `ipc.js` validates → `coordinator.js` builds the envelope (`context.js`, reading things via `app/world/things.js`), ensures a Hermes session (`gateway.request('session.create'|…)`), persists the user entry (`conversations.js`), submits `prompt.submit`, then streams: gateway frame → `events.js` normalize → `redact.js` → `conversations.js` (persist projection) → `ipc.js` push `assistant:event` → both renderers.

Per-module contract (each answers: why / owner process / public interface / must-not / callers / P1 behavior adapted):

**7.1 `app/hermes/paths.js`** — *Why:* one place that knows where Hermes lives on disk, mirroring P1's `PapersPaths` so the layout is proven. *Owner:* main. *Interface:* `hermesPaths() → {root, data, hermesHome, hermesInstall, runtime, staging, logs, hermesExecutable(), lockPath()}`; root = `PAPERS_HERMES_ROOT` env, else `path.join(process.env.LOCALAPPDATA || app.getPath('appData'), 'Papers')`; sub-layout byte-identical to `paths.rs:39–52` (`data/hermes`, `runtime/hermes-agent`, `runtime`, `staging`, `logs`); executable = `<install>/venv/Scripts/hermes.exe` (`paths.rs:70–75`); `ensure()` creates all dirs. *Must not:* touch `worldDir` (Papers world data stays where it is, `app/main.js:17–19`) or P1's `data/papers.db`/`versions.json` (never read/write them). *Called by:* runtime.js, coordinator. *P1 adapted:* `paths.rs:28–75`. **Proposal note:** using `%LOCALAPPDATA%\Papers` (same root as P1) means a machine that already ran Papers 1 reuses its verified Hermes install and Nous sign-in — deliberate reuse, see §9.3/§24 R8 for the coexistence risk and mitigation.

**7.2 `app/hermes/hermes.lock.json`** — *Why:* pinning is the install-integrity contract. Copied field-for-field from P1's `hermes.lock.json` (same version 0.18.0, tag, commit, installer URL + SHA-256, computerUse block retained for the deferred phase, protocol, upgradePolicy). *Owner:* committed file, read by main. *Must not:* be writable at runtime; never auto-updated.

**7.3 `app/hermes/runtime.js`** — *Why:* Hermes lifecycle (install/start/stop/status/health) is main-process work with secrets (token) and child processes. *Owner:* main. *Interface:* `class HermesRuntime { constructor({paths, lock}) ; async status() → BootstrapStatus ; async install() ; async start() ; stop() ; gatewayUrl() → string (throws when not running) ; async startNousLogin(onCode) → message ; async providerStatus() → sanitized {provider, model, authenticated, runtimeReady, message} ; async runEchoTest(gateway) }` — `BootstrapStatus` mirrors P1's shape `{installed, running, connected, phase, packageVersion, releaseTag, hermesHome, installDir, message}` (`models.rs:4–16`) minus `ws_url` (the URL never leaves main). *Must not:* expose the token or port through any return value that crosses IPC; parse Hermes output beyond documented endpoints; retry installs in a loop. *Called by:* coordinator, ipc.js. *P1 adapted:* `runtime.rs` almost in full (§9 maps it line-by-line).

**7.4 Active-Backpack observation (2-line change in `app/main.js`)** — *Why:* the drawer's default scope for *new* conversations needs the currently active Backpack; the renderer already tells main when a room is entered. *Proposal:* in `room:get` set `assistant.setActiveBackpack(roomId)` (that handler already calls `markRoomEntered`, `app/main.js:40–43`); in `world:get` set `assistant.setActiveBackpack(null)` (called by `loadWorld()` on entering the world view, `app/ui/app.js:51–63`, and by `backToWorld`, `86–95`). *Must not:* be used for anything after a conversation's scope locks (invariant 7).

**7.5 `app/hermes/gateway.js`** — §11 specifies it fully as a state machine. *Owner:* main. *Interface:* `class HermesGateway { connect(url) ; close(reason?) ; request(method, params, timeoutMs) → Promise ; onEvent(fn) ; onState(fn) ; state }`. Uses the Node/Electron built-in `WebSocket` client (Electron 43 ⇒ Node ≥ 22, global WebSocket available; **Risk** if unavailable in the packaged runtime — fallback is a tiny raw-socket client or promoting `ws` to a production dep; validate in Phase 3 first task). *Must not:* interpret event semantics (that's events.js) or persist anything. *P1 adapted:* `src/gateway.ts:18–160` (correlation, timeout, pending rejection) + `gateway_proxy.rs:30–123` (ping/pong is handled by the WS implementation itself; close-reason surfacing).

**7.6 `app/hermes/redact.js`** — *Why:* invariant 11. *Interface:* `isPrivateReasoningEvent(type)`, `redactPrivateReasoningFields(value)` (recursive, in-place-safe copy), `sanitizeEvent(event) → event|null`. Exact port of P1 `storage.rs:566–606`: drop event types with prefixes `reasoning.`/`thinking.`/`chain_of_thought.`; strip keys `reasoning`, `thinking`, `chain_of_thought`, `chainOfThought`, `cot` recursively. *Called by:* events.js (before anything is persisted or pushed to a renderer — stricter than P1, which redacted only at persistence).

**7.7 `app/hermes/events.js`** — *Why:* one translation point from Hermes vocabulary to Papers vocabulary, so no gateway noun leaks into UI or storage (the same containment idea as the engine seam, `app/engine/index.js:10–11`). *Interface:* `normalizeEvent(frameParams, runContext) → AssistantEvent | null` where `AssistantEvent = {kind, conversationId, runId, seq, at, data}` and `kind ∈ {run-started, text-delta, text-complete, tool-started, tool-progress, tool-completed, approval-requested, clarify-requested, run-failed, ...}` (full mapping table in §11.4). Includes ports of `payloadText` (`use-agent.ts:44–47`), `readableToolName` (`54–59`), `toolDetail` (`60–84`), `artifactDetail` (`85–94`). *Must not:* drop unknown event types silently without a debug log; invent fields.

**7.8 `app/hermes/conversations.js`** — §12 specifies the model. *Owner:* main; storage under `<worldDir>/assistant/`. *Interface:* `class AssistantStore { list() ; create({scope, title}) ; get(id) ; appendEntry(id, entry) ; bindHermesSession(id, hermesSessionId) ; lockScope(id) ; setStatus(id, status, {runId?, error?}) ; setResources(id, resources) ; delete(id) }` using the same `readJson`/atomic `writeJson` idiom as the world store (`app/world/store.js:30–43`). *Must not:* write into `rooms/**`; store unredacted payloads; allow scope mutation after lock (no API exists for it). *P1 adapted:* the `sessions` + `events` tables and state whitelist of `storage.rs:24–42, 101–293` translated to P2's JSON-file idiom.

**7.9 `app/hermes/context.js`** — §13 specifies the envelope. *Interface:* `buildEnvelope({store, conversation, instruction, resources}) → {envelope, text, refusals}` (deterministic serialization; honest per-resource share detail). *Must not:* read anything not explicitly listed in §13.2; exceed limits silently. *Reuses:* `checkThing`, `readThingContent`, `MAX_TEXT_BYTES` (`app/world/things.js`), and the honest-detail phrasing of `app/main.js:165–177`.

**7.10 `app/hermes/coordinator.js`** — *Why:* someone must own run lifecycle, session mapping, and fanout; keeping it out of `main.js` keeps `main.js` a thin registrar (its current character). *Owner:* main. *Interface:* `createAssistant({app, worldStore, worldDir}) → {ipcApi, setActiveBackpack(id|null), shutdown(), onEvent(fn)}`; internally: ensures runtime state, connects gateway on demand, maps `conversationId ↔ hermesSessionId ↔ activeRunId`, enforces one active run per conversation, applies §11 stale/duplicate rules, persists projections, pushes `assistant:event` to all subscribed webContents. *Must not:* touch `rooms/**` other than *reading* room.json/things via WorldStore for envelopes; block the world IPC handlers (all its work is async). *P1 adapted:* the orchestration half of `use-agent.ts` (96–725) moved out of the view layer into main — the key architectural translation of this plan.

**7.11 `app/hermes/ipc.js`** — §10 specifies every channel. *Why:* keep channel names/validation in one reviewable file. *Interface:* `registerAssistantIpc(ipcMain, coordinator)`, `subscribe(webContents)`. *Must not:* pass through un-validated renderer input (every handler validates shapes/lengths per §10.3).

**7.12 Renderer `app/ui/assistant.js` (+ `#assistant` root in `index.html`)** — §14. *Why a separate root:* P2's `render()` wipes `#app` innerHTML wholesale (`app/ui/app.js:1149–1152`); streaming state must live outside it. *Must not:* touch `#app` internals or `window.papers`; hold any secret.

**7.13 Companion (`app/ui/companion.html`, `companion.js`, `app/preload-companion.js`, window creation + `globalShortcut` in `main.js`)** — §15. *P1 adapted:* window config from `tauri.conf.json` companion entry, toggle/position from `main.rs:339–350, 650–662`, UI affordances from `Companion.tsx` — but state sync replaces P1's renderer-to-renderer `BroadcastChannel` with main-process fanout, because in Papers 3 the coordinator (main) is the single source of truth, whereas P1's source of truth was the main window's React state (`App.tsx:159–183`). This avoids the P1 wart where the companion is blind until the main window boots.

**7.14 Setup state (`<worldDir>/assistant/setup.json`)** — §17.3: `{schema, autoStart, lastEchoTest: {passed, at, message} | null, installNotice}` — Papers-owned, no secrets.

**7.15 Lifecycle & shutdown (wiring in `app/main.js`)** — on `app.before-quit`: `assistant.shutdown()` = best-effort `session.interrupt` for active runs (500 ms budget), `gateway.close('papers-quitting')`, `runtime.stop()` (kill child, port of `main.rs:605–610` + `runtime.rs:363–375`); `window-all-closed` keeps its existing `app.quit()` (`app/main.js:503–505`). Single-instance behavior is unchanged; because the second instance exits before `whenReady` work (`app/main.js:479–482`), it never spawns a second Hermes.

**7.16 Security boundaries** — §18. Summary: token/port/child handles live only in main; renderers get a fixed, validated `assistant:*` API; companion preload exposes strictly less; CSP unchanged (`'self'` only — the renderer never talks to Hermes directly, so no CSP loosening is needed, unlike P1 whose webview needed `ws://127.0.0.1:*`).

## 9. Runtime installation and lifecycle

This section is the Node translation of P1's proven runtime manager. Every numbered behavior cites the P1 code that proves it.

**9.1 Version lock.** `app/hermes/hermes.lock.json` pins packageVersion `0.18.0`, releaseTag `v2026.7.1`, commit `7c1a029553d87c43ecff8a3821336bc95872213b`, installerUrl (raw.githubusercontent.com under that exact commit), installerSha256 `575a1504…62bcd`, plus the computerUse block (retained, unused in v1) and `upgradePolicy: "manual-after-compatibility-check"` (**Observed:** P1 `hermes.lock.json`). Loaded once at construction; malformed lock disables the assistant with an honest message but never blocks Papers (softening of `runtime.rs:36–38`, justified by invariant 4).

**9.2 Installer integrity.** Download the pinned URL with `https` through Node; compute SHA-256 via `node:crypto`; compare case-insensitively to the lock; on mismatch, delete nothing, run nothing, message: "Hermes installer verification failed. Expected <lock>, received <actual>. Nothing was run." (**Observed:** `runtime.rs:164–170`; the mismatch path was really exercised once — P1 `PROJECT_CONTEXT.md` "Papers rejected an incorrect Hermes installer checksum and executed nothing"). Save the verified bytes to `<root>/runtime/hermes-install.ps1` (`runtime.rs:151, 171–172`).

**9.3 Install directory & user-data directory.** Root `%LOCALAPPDATA%\Papers` (override `PAPERS_HERMES_ROOT`; P1 used `PAPERS_DATA_HOME`, renamed to avoid ambiguity with P2's `PAPERS_WORLD_DIR`): install at `<root>/runtime/hermes-agent`, HERMES_HOME at `<root>/data/hermes`, logs `<root>/logs`, staging `<root>/staging` (**Observed:** `paths.rs:39–52`). The Papers *world* stays at its existing location (`app/main.js:17–19`) — two roots, two owners, no mixing. Sharing P1's root deliberately reuses an existing verified install + sign-in when present (§7.1); the coexistence risk (both apps running one Hermes home) is R8 in §24 — mitigation: Papers 3 always starts its *own* `hermes serve` on its own port; HERMES_HOME contention is limited to config/auth files that Hermes itself manages, and the user is one person on one machine.

**9.4 Executable discovery.** `<install>/venv/Scripts/hermes.exe`; `installed` = that file exists (**Observed:** `paths.rs:70–75`, `runtime.rs:76–77`). Never search PATH; never run any other hermes.

**9.5 Partial installs.** If the install dir exists without the executable, rename it to `<root>/staging/hermes-agent-partial-<millis>` before reinstalling — preserve, don't delete (**Observed:** `preserve_partial_install`, `runtime.rs:242–257`).

**9.6 Installer process.** `powershell.exe -NoLogo -NoProfile -ExecutionPolicy Bypass -File <installer> -Commit <commit> -HermesHome <hermesHome> -InstallDir <install> -SkipSetup -NonInteractive`, env `HERMES_HOME=<hermesHome>`, `GIT_CONFIG_GLOBAL=<data>/installer.gitconfig` (file content exactly `[core]\n\tautocrlf = false\n[windows]\n\tappendAtomically = false\n`), `PLAYWRIGHT_BROWSERS_PATH=<root>/runtime/playwright`; PATH filtered to exclude entries containing `\windowsapps`, `\chocolatey\bin`, `\scoop\shims` (case-insensitive, `/`→`\` normalized); stdout+stderr appended to `logs/hermes-install.log` with a timestamp separator line; `windowsHide: true` (**Observed:** `runtime.rs:177–228, 259–272, 925–953`; `hide_console` `935–938`). Before launching, snapshot user-scope `HERMES_HOME` and `Path` via `powershell [Environment]::GetEnvironmentVariable(...,'User')` and restore both afterwards, reporting restore failure honestly (**Observed:** `runtime.rs:178–216, 955–969, 996–1021`). Post-conditions: executable exists, else fail with "installer finished without producing its executable" (`runtime.rs:230–235`); then write the guarded config (9.7). If already installed, `install()` is idempotent: guarded config + status (`runtime.rs:142–146`).

**9.7 Guarded Hermes config (safety-critical).** Before every start and after install, ensure `HERMES_HOME/config.yaml` contains: `approvals: {mode: manual}`, `toolsets: ["hermes-cli"]` (v1 — P1 also had `computer_use`, deferred), `agent.yolo: false` if absent (**Observed:** `write_guarded_config`, `runtime.rs:738–770`; builder-profile half `771–808` not ported). Node implementation: parse with the `yaml` package; if the existing file fails to parse, **refuse to start** with "Papers could not read Hermes' settings at <path> and will not overwrite them. Fix or remove the file, then try again." (stricter than P1, which fell back to `{}` and would clobber — `runtime.rs:741–748`; the stricter rule protects a hand-edited Hermes home). Writes are atomic tmp+rename (same idiom as `write_main_config`, `runtime.rs:588–598`, and P2's `writeJson`, `app/world/store.js:38–43`). Provider/model changes use the same read-patch-write path (`model` map: `default`, `provider`; strip stale `api_key`/`base_url`/`api_mode`/`auth_mode` keys — **Observed:** `runtime.rs:501–532`) with the same normalization rules (`normalize_provider`/`normalize_model`, `runtime.rs:876–900`).

**9.8 Port selection.** Bind a `net.Server` to `127.0.0.1:0`, read the assigned port, close the server, use the port (**Observed:** `available_port`, `runtime.rs:844–849`). The bind-then-release race is accepted exactly as P1 accepts it; a failed serve start surfaces honestly (§19 row 5).

**9.9 Process arguments & environment.** Spawn `<hermes.exe> serve --host 127.0.0.1 --port <p>` with `cwd = <root>/runtime` and env: `HERMES_HOME`, `HERMES_EXEC_ASK=1`, `HERMES_DASHBOARD_TUI=1` (mandatory — without it Hermes closes the WebSocket immediately; **Observed:** P1 `PROJECT_CONTEXT.md`), `HERMES_DASHBOARD_SESSION_TOKEN=<token>`, `PLAYWRIGHT_BROWSERS_PATH=<root>/runtime/playwright`; stdout/stderr appended to `logs/hermes-serve.log`; `windowsHide: true`; **not** `detached` (the child must die with Papers) (**Observed:** `runtime.rs:311–330`). `HERMES_CUA_DRIVER_CMD` omitted in v1 (matching the reduced toolset in 9.7).

**9.10 Local host binding.** `--host 127.0.0.1` only; the health client and gateway URL also hardcode `127.0.0.1` (**Observed:** `runtime.rs:313, 561, 813`). Papers 3 must never widen this.

**9.11 Authentication token.** Two concatenated UUIDv4s generated per start via `crypto.randomUUID()` (**Observed:** `runtime.rs:305`); held only in `HermesRuntime`'s private state; used in the env var (9.9) and in the gateway URL query (`?token=`); cleared on stop; never logged, never crosses IPC, never in any status object (§18).

**9.12 Health checks, startup timeout, retries.** After spawn: poll `GET http://127.0.0.1:<port>/api/status` every 250 ms up to 120 attempts (30 s), simultaneously checking child liveness; child exit ⇒ fail "Hermes stopped while starting. See <log>."; timeout ⇒ fail "Hermes did not become ready within 30 seconds. See <log>." (**Observed:** `runtime.rs:338–360`). `status()` recomputes `{installed, running, connected, phase}` on demand, treating a reaped child as stopped and clearing port/token (**Observed:** `runtime.rs:74–139`). No automatic install retries; start retries only on explicit user action, except the single crash-restart in §11.8.

**9.13 Shutdown & stale processes.** `stop()` kills the child (SIGKILL semantics via `child.kill()`, then handle `close`), clears port/token (**Observed:** `runtime.rs:363–375`). Wired to `before-quit` (§7.15; P1 equivalent `main.rs:605–610`). Stale-process handling: because the port and token are per-launch and never persisted, a previous Papers crash can leave an orphan `hermes serve`; Papers 3 never adopts an unknown process — it starts a fresh one on a fresh port (orphans idle without a token holder). **Deferred hardening:** a `logs/hermes-serve.pid` file plus best-effort taskkill of a stale pid at startup (documented in §19 row "stale process"; not required for correctness).

**9.14 Duplicate Papers instances.** Unchanged single-instance gate (`app/main.js:479–482`) means at most one Papers, therefore at most one Papers-owned Hermes per profile.

**9.15 Crashes.** Child `exit` while `running` ⇒ coordinator transitions per §11.8 (gateway closes at the same time); the run in flight is marked `interrupted`; the drawer shows §6.18. Logs stay in `logs/hermes-serve.log` with timestamped section separators (**Observed:** `create_log`, `runtime.rs:925–933`).

**9.16 Logs & safe diagnostics.** Papers-owned logs: `hermes-install.log`, `hermes-serve.log` (append, timestamp-separated). The drawer's failure states offer **[Open log folder]** → `shell.openPath(<root>/logs)` (P2 already uses `shell.openPath`, `app/main.js:94`). Diagnostics rule: log messages may contain paths and exit codes, never the token and never prompt/response content.

**9.17 Windows specifics.** All commands run with `windowsHide: true` (P2 precedent `app/engine/runtimes/claude-cli.js:28`; P1 `CREATE_NO_WINDOW`); PowerShell invoked as `powershell.exe` (not pwsh); paths built with `path.join` throughout; the venv executable layout is Windows-only — on non-Windows dev machines `status()` reports `not_installed` with the honest message "Papers' private Hermes runtime is currently Windows-only", and all tests that need Hermes use the fake (§20).

**9.18 Upgrade policy.** Manual only, per the lock. Changing the lock is a reviewed commit + full §20 re-run. No auto-update code exists.

### 10.2 Pushed events (main → renderer, `webContents.send('assistant:event', evt)`)

One channel, one envelope: `{kind, conversationId?, runId?, seq, at, data}`. Kinds: `gateway-state` (`{state, reason?}` — §11 states), `setup-progress` (`{phase, detail}`), `run-started`, `text-delta` (`{text}`), `text-complete` (`{text}`), `tool-started`/`tool-progress`/`tool-completed` (`{toolId, name, detail?, artifact?}`), `approval-requested` (`{approvalId, command, description}`), `clarify-requested` (`{requestId, question, choices}`), `run-completed`, `run-failed` (`{error}`), `run-interrupted`, `conversation-updated` (`{conversationId}` — list refresh hint), `status-changed` (`{status}`). Preload wraps subscription as `papersAssistant.onEvent(fn) → unsubscribe` using a module-scoped `ipcRenderer.on` — the raw `ipcRenderer` is never exposed. All payloads pass `redact.js` before send (stricter than P1, §7.6).

### 10.3 Security validation (every handler, in `ipc.js`)

* `conversationId`/`runId`/`thingId`/`roomId` must match `/^[a-z]+_[0-9a-f]{16}$/` (the `newId` shape, `app/world/ids.js:5–7`) or the P1-style UUID for Hermes-side ids; unknown ids ⇒ honest `{ok:false}`.
* `text` ≤ 32,000 chars; `title` ≤ 140; `answer` ≤ 8,000; arrays ≤ 16 resources.
* `resources[].path` (Global scope) must be an absolute path that was returned by the immediately preceding `assistant:pickResources` dialog for that conversation (main keeps the last dialog result per conversation; the renderer cannot mint arbitrary paths — path-traversal defense, §18).
* `scope.roomId` must name an existing room (checked via `WorldStore.getRoomRecord`, which throws honestly — `app/world/store.js:115–119`).
* Senders: events are pushed only to the two known webContents (main window, companion) registered by main.js at window creation; `ipc.js` checks `event.sender` identity on every invoke and refuses others.

### 10.4 Timeout & cancellation

Gateway-bound calls inherit §11 request timeouts (`session.create`/`session.resume` 60 s, `prompt.submit` 1,800 s, others 120 s default — P1 values, `gateway.ts:66`, `use-agent.ts:474–518, 645–649`). `assistant:interrupt` is the only cancellation primitive, mapping to `session.interrupt` (P1 `use-agent.ts:558–567`). IPC handlers themselves never hang: they resolve `{ok:false, error}` when the coordinator reports a state that cannot accept the request (§11.3 table).

### 10.5 Companion subset (`app/preload-companion.js`)

Exposes only: `status`, `listConversations`, `getConversation`, `selectConversation`, `send` (text only — the handler additionally rejects `resources` from the companion sender), `interrupt`, `openMain`, `hideCompanion`, `onEvent`. No setup, no install, no provider calls, no resource picking, no delete/rename. Least privilege by construction.

## 11. Hermes gateway adapter (explicit state machine)

### 11.1 States and transitions

```
                       install()            start()
  unavailable ──────► installing ──────► starting ──────► connecting ──────► ready
   ▲   ▲                  │ fail             │ fail            │ fail          │
   │   └──────────────────┘                  ▼                 ▼               │
   │                                       failed ◄────────────┘               │
   │                                         ▲                                 │
   │            stop()/quit                  │ retries exhausted               │
   └────────── stopping ◄────────────────────┼──────────────── reconnecting ◄──┘
                                             └──────────────────┘  socket lost
```

| State | Meaning | Entered by | Exits to |
|---|---|---|---|
| `unavailable` | not installed (or lock invalid) | boot when `!installed` (`runtime.status()`, §9.12) | `installing` |
| `installing` | verified install running | `assistant:install` | `starting` (auto), `failed` |
| `starting` | child spawned, health loop running | `assistant:start` / auto-start (§6.21) / crash-restart (11.8) | `connecting`, `failed` |
| `connecting` | WS dialing `ws://127.0.0.1:{port}/api/ws?token=…` | health OK | `ready` (on socket open **and** first frame or 2 s grace — P1 treats open as connected, `gateway.ts:36–44`; Papers 3 also accepts the observed `gateway.ready` event as confirmation, `use-agent.ts:196–198`), `failed` (15 s dial timeout — P1 value, `gateway_proxy.rs:32–38`) |
| `ready` | requests allowed | socket open | `reconnecting` (socket lost, child alive), `stopping`, `failed` (child died) |
| `reconnecting` | bounded redial: 3 attempts, 1 s/2 s/4 s backoff | socket close while child alive | `ready`, `failed` |
| `stopping` | graceful teardown (§7.15) | `assistant:stop` / quit | `unavailable`/`ready-to-start` (i.e. installed+stopped) |
| `failed` | terminal until user action | any failure | `starting` (user [Try again]/[Start]), `installing` |

The runtime `phase` strings surfaced to the UI (`not_installed, installing, stopped, starting, ready, error` — P1 `types.ts:12–18`) are a projection of these states plus `installed`.

### 11.2 Transition triggers (exact)

* health loop success (§9.12) ⇒ `starting → connecting`.
* WS `open` ⇒ `connecting → ready`; push `gateway-state {state:'ready'}`.
* WS `close`/`error` while child alive ⇒ `ready → reconnecting`; all pending requests reject with the close reason (**Observed:** P1 rejects pending on close — `gateway.ts:105–113, 146–152`; close-reason surfacing from `gateway_proxy.rs:70–87`).
* child `exit` ⇒ from any state ⇒ `failed` (crash path 11.8) or `stopping` (if we initiated).
* `stop()` ⇒ `stopping` ⇒ installed+stopped.

### 11.3 Allowed requests per state

| State | Allowed | Everything else returns |
|---|---|---|
| `ready` | all six requests | — |
| `reconnecting`/`connecting`/`starting` | none (queueing is **not** done — P1 rejects immediately with "Hermes is not connected", `gateway.ts:69–71`; Papers 3 keeps that honesty) | `{ok:false, error:'Hermes is not connected yet — <state>.'}` |
| others | none | `{ok:false, error:<state-appropriate message + recovery action>}` |

### 11.4 Event parsing & normalization

Frames are parsed exactly as P1 does (`gateway.ts:116–144`): JSON parse failure ⇒ drop + increment a `malformedFrames` counter (surfaced in the drawer only after 5 in a session: "Hermes sent unreadable data; see log"); frames with `id` are responses correlated against the pending map (unknown id ⇒ drop — duplicate-response protection); frames with `method === "event"` and `params.type` fan out. Normalization mapping (events.js), from the observed handler (`use-agent.ts:194–351`):

| Hermes event `type` | AssistantEvent kind | Notes (observed payload usage) |
|---|---|---|
| `gateway.ready` | `gateway-state {state:'ready'}` | connection confirmation (`use-agent.ts:196–198`) |
| `message.start` | `run-started` + open text buffer | (`199–214`) |
| `message.delta` | `text-delta {text}` | text = `payload.text ?? payload.rendered ?? payload.message` (`44–47, 215–229`); empty ⇒ dropped |
| `message.complete` | `text-complete {text}` + `run-completed` | prefer final text over accumulation (`230–253`) |
| `tool.start` / `tool.progress` / `tool.generating` | `tool-started`/`tool-progress` | humanized name; detail from `payload.description` (`255–270`) |
| `tool.complete` | `tool-completed {name, detail, artifact?}` | detail via `toolDetail` (`60–84`), artifact via `artifactDetail` (`85–94, 271–293`) |
| `approval.request` | `approval-requested {approvalId(new), command, description}` | fields `payload.command`, `payload.description`, `payload.allow_permanent` (`294–318`); status → `awaiting_approval` |
| `clarify.request` | `clarify-requested {requestId, question, choices[]}` | requires `payload.request_id` and `payload.question` (`320–338`) |
| `error` | `run-failed {error}` | message via payloadText, fallback "Hermes reported an unknown error." (`340–347`) |
| `reasoning.*`, `thinking.*`, `chain_of_thought.*` | **dropped before normalization** (redact.js) | P1 storage precedent (`storage.rs:576–581`); Papers 3 also never renders them |
| any other `type` | `null` (ignored) + debug log | forward-compatibility rule: unknown events must not crash or render (`use-agent.ts:349–351`) |

### 11.5 Request correlation & session mapping

JSON-RPC `id` is a per-connection incrementing integer (`gateway.ts:21, 73`); the pending map stores `{resolve, reject, timer}`; timers use the per-method timeouts of §10.4. Session mapping lives in the coordinator: `conversation.hermesSessionId` is set from `session.create`'s `stored_session_id ?? session_id` (**Observed:** `use-agent.ts:481–496`) and refreshed by `session.resume`'s returned `session_id` (which may differ — `use-agent.ts:650–653`). Events are attributed by `params.session_id`: an event carrying a session_id that maps to no known conversation is dropped (P1 drops non-active-session events, `use-agent.ts:185–187`; Papers 3 generalizes to a map because multiple conversations exist, but only one run may be active per conversation).

### 11.6 Run correlation, ordering, partial streams

**Observed:** the Hermes protocol as used by P1 has no run id; a run is implicitly "everything on this session between `prompt.submit` and its terminal event." Papers 3 therefore synthesizes `runId = newId('run')` at submit; events for that session are stamped with the active runId; `text-delta` with no open run is dropped (P1's null-check, `use-agent.ts:217–220`). Ordering: WebSocket delivery order per connection, re-sequenced with a per-conversation `seq` counter at persist time. Terminal events for a run: `text-complete`→`run-completed`, `run-failed`, `run-interrupted` (local, after a successful `session.interrupt`), or `gateway-state: closed` (⇒ `run-interrupted`, reason "connection lost"). A partial stream (deltas then close, no complete) persists whatever streamed plus a `status` entry "interrupted — the reply above may be incomplete."

### 11.7 Duplicate & stale protection

Duplicate responses: unknown/already-settled ids dropped (11.4). Duplicate events: Hermes was not observed to duplicate, but replays after `session.resume` are possible in principle; guard: while a resume is in flight, incoming events for that session are ignored until the resume response settles (P1's resume also rebuilds message state wholesale, discarding stream state — `use-agent.ts:654–681`). Stale events: any event whose session maps to a conversation whose `activeRunId` is null and whose kind is delta/tool ⇒ dropped and counted.

### 11.8 Process-loss handling

Child exit detected (child `exit` event or reaped by `status()`): mark gateway `failed`, reject pending, mark in-flight runs `run-interrupted (Hermes stopped)`. If the exit was not user-initiated and Papers was in `ready`, attempt exactly one automatic restart cycle (`start()` → reconnect); if that fails, stay `failed` with §6.18 UI. Never loop (invariant 12; P1 has no auto-restart at all — the single retry is Papers 3's one liveness improvement, bounded deliberately).

## 12. Conversation and scope model

### 12.1 Data model (persisted per conversation, schema v1)

`<worldDir>/assistant/conversations/<conversationId>.json`:

```jsonc
{
  "schema": 1,
  "id": "conv_9f2c40a1b7d3e654",           // newId('conv') — app/world/ids.js
  "title": "Plan the gate rebuild",          // first user message slice(0,70) — P1 rule use-agent.ts:459–461; renameable
  "scope": { "kind": "global" },             // or { "kind": "backpack", "roomId": "room_…", "titleAtLock": "Field notes" }
  "scopeLockedAt": null,                     // ISO — set at first successful prompt.submit; null = still changeable
  "hermesSessionId": null,                   // bound after session.create — stored_session_id ?? session_id
  "createdAt": "…", "updatedAt": "…",
  "status": "idle",                          // idle|running|awaiting_approval|awaiting_clarification|interrupted|failed
  "activeRunId": null,                       // run_… while running
  "resources": [],                           // last-attached chips (per-message send list is in each entry)
  "lastError": null,                         // {message, at} | null
  "entries": [                               // redacted local projection (§12.8)
    {"id":"entry_…","seq":1,"role":"creator","text":"…","at":"…",
     "resources":[{"displayName":"plan.txt","path":"…","shareDetail":"3,412 of 3,412 bytes"}]},
    {"id":"entry_…","seq":2,"role":"assistant","text":"…","at":"…","meta":{"runId":"run_…"}},
    {"id":"entry_…","seq":3,"role":"tool","text":"Command finished","detail":"…","at":"…"},
    {"id":"entry_…","seq":4,"role":"status","text":"You stopped this run.","at":"…"}
  ]
}
```

Status values are a whitelist validated on write (P1 precedent: `storage.rs:237–261`). Roles reuse P2's conversation vocabulary (`creator`/`ai`→`assistant`/`status` — `app/world/store.js:521–533`) plus `tool`.

### 12.2 When scope locks, and how it is enforced

* **Chosen:** at creation — default `backpack:<activeBackpackId>` when created inside a Backpack, `global` otherwise (§6.6–6.7). Changeable via `assistant:setScope` while `scopeLockedAt === null` **and** `entries` is empty.
* **Locks:** in `coordinator.send`, immediately **before** the `prompt.submit` request is issued for the conversation's first message; on submit failure the lock stands (the user's intent was expressed; envelope already built against that scope). `scopeLockedAt` set; for Backpack scopes `titleAtLock` snapshots the room title (survives later deletion, §12.6).
* **Enforced:** (a) storage: `AssistantStore` exposes no scope mutation after lock, and `setScope` on a locked conversation returns the honest error (§10.1); (b) behavior: after lock the coordinator resolves everything (envelope, listing group, chip) from `conversation.scope` only — `setActiveBackpack` is consulted solely for *defaults of new conversations* (invariant 7); (c) tests pin both (§20.1).

### 12.3 How scope is displayed

Chip in the drawer header and companion (§6.6–6.7, §6.14): "Global — this whole world" / "Backpack: <current room title, refreshed from room.json each render>" / "Backpack no longer in this world: <titleAtLock>" (deleted case). Before lock, the chip carries the toggle affordance; after lock, it is static text plus tooltip "Scope locks with the first message and cannot change."

### 12.4 Listing & selection

`assistant:listConversations` returns metas sorted by `updatedAt` desc (P1 ordering, `storage.rs:106–107`). The drawer groups: **This Backpack** (scope.roomId === activeBackpackId), **Global**, **Other Backpacks** (each labeled with its room title or `titleAtLock`). Drawer and companion both select via `assistant:selectConversation`; the coordinator holds one `selectedConversationId` shared by both windows, so selecting in either surface moves both (single projection — product constraint §"Global Hermes access"; fanout via `conversation-updated`/`status-changed` events).

### 12.5 Active-Backpack changes

Entering/leaving Backpacks calls `setActiveBackpack` (§7.4). Effects: regroup the list; retitle the "This Backpack" group; update the §6.14 notice line. **Non-effects (tested):** no conversation switch, no scope change, no interrupt, no re-context of in-flight or future messages of existing conversations.

### 12.6 Deleted / renamed / unavailable Backpack

* **Renamed:** scope key is `roomId`; display reads the live title (rename already logs a room event — `app/world/store.js:121–130`). Nothing else changes.
* **Deleted:** P2 today has no room-delete flow (**Observed:** no `deleteRoom` in `store.js`, no IPC), so this is future-proofing: if `getRoomRecord` throws for the scope's roomId, display the deleted-chip (§12.3); envelope replaces the Backpack block with the honest line "This conversation is scoped to a Backpack that is no longer in this world (\"<titleAtLock>\")."; sends remain allowed; Backpack-things resource picking is disabled (native dialog still allowed).
* **Unavailable (corrupt room.json):** same handling as deleted, message "…cannot currently be read."

### 12.7 Resumability

On select (when `ready` and `hermesSessionId` present): `session.resume {session_id, cols:96}` (60 s). Success: rebind possibly-new `session_id`; do **not** overwrite the local projection with Hermes' `messages[]` — the local projection is already richer (tool/status entries); instead reconcile only the banner (§6.16). Failure (error response / timeout): keep the projection, clear `hermesSessionId` (a fresh session will be created on next send — same recovery as P1's unbound sessions, which simply create anew, `use-agent.ts:468–470`), banner per §6.16. **Papers-owned:** the projection, scope, status. **Hermes-owned:** true session memory/history, tool state, provider auth.

### 12.8 Why a local projection (justified duplication)

The drawer must render history when Hermes is stopped/uninstalled (invariant 3/12), and `session.resume` returns only user/assistant text and only when Hermes runs (**Observed:** `use-agent.ts:657–681`). P1 faced the same need and persisted redacted events in SQLite (`storage.rs:263–293`). Papers 3 persists the *rendered projection* (entries), not raw event streams — smaller, human-legible (P2 idiom), and free of protocol vocabulary. Raw frames are never persisted.

## 13. Context envelope

### 13.1 Shape (built per message by `context.js`)

```jsonc
{
  "version": 1,
  "scope": "global" | "backpack",
  "backpack": {                       // present only for backpack scope and only if the room is readable
    "id": "room_…",
    "title": "Field notes",           // live room.json title
    "description": "…"                // room.description, capped 500 chars, omitted when empty
  },
  "resources": [                      // explicitly attached this message; max 8; attach-order preserved (deterministic)
    {
      "displayName": "plan.txt", "path": "C:\\…\\plan.txt", "type": "file", "status": "present",
      "content": { "kind": "text", "shownBytes": 3412, "totalBytes": 3412, "truncated": false, "text": "…" }
      // kinds and caps exactly readThingContent: text ≤ MAX_TEXT_BYTES (24 KiB, app/world/things.js:11),
      // folder-listing ≤ 200 entries, binary → name+size only, unreadable → honest error line
    }
  ],
  "instruction": "the user's message text"
}
```

### 13.2 Serialization to the prompt

Hermes' `prompt.submit` takes plain `text`; P1 prepends context as text (`use-agent.ts:498`). `renderEnvelopeText(envelope)` produces, deterministically (fixed section order, fixed wording, attach-order resources):

```
[Papers context — provided explicitly by the creator]
Scope: the Backpack "Field notes" in Papers. |or| Scope: global — the creator's whole Papers world.
Backpack description: …            (when present)
Attached resource 1 of N: plan.txt (file, present, at C:\…\plan.txt)
=== content, read just now; truncation and unreadable content are marked honestly ===
…
[End of Papers context]

<instruction>
```

The wording mirrors P2's prompt honesty conventions (`app/engine/prompts.js:82–101`). Narrow system guidance is limited to the bracketed header lines — no persona construction, no behavioral essay (Hermes owns its own agent behavior).

### 13.3 Limits & truncation

Per resource: `readThingContent` caps (24 KiB text / 200 folder entries) with its built-in truncation markers (**Observed:** `app/world/things.js:58–110`). Per envelope: ≤ 8 resources, rendered text ≤ 192 KiB; exceeding either returns a refusal to the drawer before anything is sent: "That's more than this message can carry — detach something or send in parts." (No silent dropping — invariant 8.)

### 13.4 Attachment semantics

Resources are attached per message via the picker (§6.8) and cleared from the composer after send; the conversation's `resources` field remembers the last set only as chips for one-click re-attach. Nothing is inferred: no auto-attach of Desk items, open notes, or recent files. Missing resources at send time are re-checked (`checkThing`) and included as their honest unreadable/missing form rather than silently dropped — the chip preview already showed the risk.

### 13.5 Unsupported / missing resources, privacy

Binary → name+size only; unreadable → error line; missing → "missing — nothing exists at this path right now" (all existing `readThingContent`/preview wording, `app/ui/app.js:610–644`). Privacy boundary: only chip-previewed content enters the envelope; Papers never sends the conversation history back (Hermes owns session memory); Backpack metadata is limited to id/title/description — never things lists, notes, Desk state, or activity (contrast with the *room* engine's `roomContextBlock`, which is a different, existing feature and stays as-is).

## 15. Companion design

Behavioral reference: P1 `Companion.tsx:1–78`, window config in `tauri.conf.json`, toggle/positioning in `main.rs:339–350, 505–531, 650–662`. Papers 3 re-expresses these in Electron; nothing is copied as code.

* **Creation:** `createCompanionWindow()` in `main.js` at startup: `new BrowserWindow({width: 460, height: 180, frame: false, alwaysOnTop: true, skipTaskbar: true, resizable: false, show: false, backgroundColor: '#f5f0e6', webPreferences: {preload: preload-companion.js, contextIsolation: true, nodeIntegration: false}})`, loads `app/ui/companion.html`. Created hidden once; never destroyed until quit (P1 keeps it alive and hidden). `closable` behavior: intercept `close` → `hide()` while the app runs.
* **Show/hide:** global `Ctrl+Alt+Q` toggles (`main.rs` semantics): if visible → hide; else position bottom-right — `screen.getDisplayNearestPoint(screen.getCursorScreenPoint()).workArea`, x = right − width − 18, y = bottom − height − 56 (the P1 margins, `main.rs:650–662`, using workArea so the taskbar is respected) — then `show()` + `focus()`. Also `assistant:hideCompanion` from its own ✕ (P1 `hideCompanion`, `papers.ts:83`).
* **Shared state:** the companion is a second subscriber to the same coordinator fanout (§10.2) — same selected conversation, same events, same status. No BroadcastChannel, no second state system (§7.13). It renders: status dot+word, active conversation title + scope chip, last streamed line or pending-attention notice, and the input.
* **Submitting prompts:** its input calls `assistant:send` on the **selected conversation**; if none exists, it first calls `assistant:createConversation` with scope defaulted by the coordinator's activeBackpack (same rule as the drawer, §12.2) and shows the scope it chose ("New Global conversation" / "New conversation in <Backpack>") so nothing is silent. Text only — no resource attachment from the companion (§10.5).
* **Displaying scope & status:** chip identical to §12.3; status word identical to the drawer (single copy map).
* **Pause/interrupt:** [Stop] → `assistant:interrupt`. (P1's Pause/Resume canned-prompt pair — `use-agent.ts:569–593` — is Deferred with §6.13's rationale.)
* **Open full Papers:** [Open Papers] → `assistant:openMain` → show/unminimize/focus main window (port of `show_main`, `main.rs:353–360`).
* **No duplicate gateway/session:** structurally impossible — gateway and sessions live in the coordinator (main); the companion holds no connection state at all. (P1 needed the main webview alive to work; Papers 3's companion works whenever main *process* is alive.)
* **When Papers is closing:** `before-quit` hides the companion immediately, then §7.15 teardown; the global shortcut is unregistered in `will-quit` (M1).
* **When no conversation exists:** shows "No conversations yet" + the input (which creates one on submit, above) + [Open Papers]; when Hermes isn't ready: status word + "Open Papers to finish setup" + [Open Papers] — the companion never exposes install/setup actions (§10.5 least privilege).

## 16. Existing AI compatibility

**Recommendation (Decision for Papers 3): keep the existing Backpack AI unchanged initially — option (a).**

Evidence for least disruption:

1. The existing engine is a deliberate, tested seam with exactly four Papers-native actions and a one-shot `complete(prompt)` contract (`app/engine/index.js:32–47, 52–87`). Its consumers are five `main.js` handlers (`room:say` 354–367, `room:createNote` 400–428, `desk:synthesize` 196–236, `note:revise` 287–314, plus previews that only *build* material). None of them can stream, and two of them (revision, desk synthesis) are contractually bound to *exact prompt bytes approved in advance* (`app/world/revision.js:37–75`; test `world.test.js:790–813`). Hermes' streaming, tool-using, session-based model is a different contract; forcing the fingerprint-bound revision flow through a stateful agent session would weaken its strongest guarantee.
2. The world tests pin engine-seam behavior extensively (`engine.test.js` all; `world.test.js:556–595` even asserts the engine is *not loaded* on preview paths). Replacing runtimes immediately would churn the most safety-relevant tested surface for zero user-visible gain.
3. The product constraint says existing Backpack behavior must keep working while Hermes is introduced, and forbids replacement "merely for architectural purity."

**Coexistence rules:**

* Two AI paths exist and are honestly labeled: Backpack notes/replies show their engine provenance as today (`meta.engine`, provenance chips — `app/main.js:361, app/ui/app.js:539`); assistant output lives only in the assistant drawer/companion. No shared state, no shared config.
* Runtime selection for the old path stays `PAPERS_ENGINE` (dev-level, `app/engine/index.js:26–30`); the assistant's provider/model is Hermes-owned (§9.7). These never read each other's configuration — duplicate model configuration is avoided by *not* configuring models twice (the old path has no product-surface model config at all; **Observed:** README "Runtime selection is a dev-level concern").
* UI confusion is addressed by naming: the room panel keeps its "Backpack conversation" title (`app.js:683`); the new surface is always "Assistant". The assistant drawer never renders inside the room conversation panel.
* Persistence compatibility: `conversation.json` untouched (§17). No entry kinds are added to it.

**Path to later unification (Deferred, gated):** (1) add a fifth engine action or a `hermes` runtime adapter implementing the existing `complete(prompt)` contract against a throwaway Hermes session — possible without touching callers because the seam is uniform; (2) run it behind `PAPERS_ENGINE=hermes` for `room:say` only; (3) compare quality/latency/failure honesty for a period; (4) only then consider notes/Desk/revision, which additionally require Hermes to honor exact-prompt-in/exact-text-out without tool use (verifiable via config `toolsets: []` per session — **Risk:** unverified against Hermes 0.18.0; must be tested before any migration). **Removal criteria for old runtimes:** never during Papers 3; revisit only after the Deferred experiment shows ≥ parity for months of real use. **Rollback:** unset `PAPERS_ENGINE` — the default runtime remains `claude-cli` (`app/engine/index.js:26`).

## 18. Security and privacy

* **Renderer isolation:** unchanged flags (`contextIsolation:true, nodeIntegration:false`, `app/main.js:453–457`; same for the companion). CSP stays `'self'`-only (`app/ui/index.html:5`; same header in companion.html) — possible only because *no* renderer ever talks to Hermes directly (contrast P1, which had to allow `ws://127.0.0.1:*` in its webview CSP, `tauri.conf.json`).
* **Preload allowlist:** two frozen namespaces (`papers` existing; `papersAssistant` §10); companion gets the §10.5 subset via its own preload. No `ipcRenderer` passthrough, no dynamic channel names.
* **Child-process ownership:** only main spawns processes; the Hermes child handle never leaves `HermesRuntime`; `windowsHide` everywhere; not detached (§9.9).
* **Local gateway binding:** `127.0.0.1` only, both serve and client (§9.10).
* **Authentication token:** §9.11 — main-process memory only; regenerated per start; in env of the child and the WS URL; excluded from status objects, IPC payloads, logs, and errors (a gateway error message must never echo the URL — gateway.js strips the query string from any error text it surfaces).
* **Secrets / provider credentials:** Hermes-owned under HERMES_HOME (`auth.json` — P1 reads only `active_provider`/provider presence, `runtime.rs:600–632`; Papers 3 identical). Never rendered beyond booleans/names; never copied into the world dir; `.gitignore` already refuses `auth.json` (P2 `.gitignore`). The isolated-key-entry window of P1 is not needed (no API-key entry in v1 scope).
* **Logs:** §9.16 rule (paths + exit codes only). Assistant entries persist redacted text but logs never include prompt/response content.
* **Prompt context:** only §13 envelope content leaves Papers; chips previewed the exact share detail first. Hermes' own tools may read more, but only under Hermes approvals, which §9.7 pins to `manual` + `yolo:false` (**Observed:** the same guard P1 writes, `runtime.rs:754–763`).
* **Resource access / path traversal:** Backpack-scope resources must be things of the scope room (validated against `listThings`); Global-scope paths must come from the immediately preceding main-process dialog result (§10.3) — the renderer can never name an arbitrary path. (P2's existing surface has the same property: paths enter only via `dialog.showOpenDialog`, `app/main.js:62–73`.)
* **Approvals:** rendered exactly (command shown verbatim in `<pre>`); responses map 1:1 to `approval.respond`; no auto-approve, no "always" in v1 (§6.11).
* **Hidden reasoning:** redact.js gate ahead of both persistence *and* renderer push (§7.6) — strictly stronger than P1, where redaction was persistence-only and the webview did receive raw frames (`gateway_proxy.rs:57`).
* **Malicious/malformed Hermes events:** all payload field reads are type-checked with fallbacks (the P1 handler's `typeof` discipline, `use-agent.ts:295–335`); unknown kinds dropped; strings length-capped before render (`esc()` everywhere — `app/ui/app.js:32–38` idiom); counters + log for malformed frames (§11.4). An event cannot trigger any Papers action other than rendering and status transitions — approval/clarify *responses* always originate from a user click.
* **Malformed renderer requests:** §10.3 validation; unknown sender webContents refused.
* **Oversized payloads:** envelope caps (§13.3); IPC input caps (§10.3); delta accumulation capped at 2 MiB per message with honest truncation entry (protects the projection file).
* **Abandoned processes:** §9.13; single-instance lock unchanged; orphan Hermes without its token holder is inert and cannot be adopted.

Scripts are declarative JSON so each test states its scenario. It can run in-process (unit-style) or as a child process with `--port`/`--token` args (lifecycle-style).

## 23. Acceptance criteria (all binary)

1. Opening Papers 3 against a pre-existing Papers 2 world directory shows identical world/room content, and after a full session the `rooms/**` tree is byte-identical except changes the user made through existing features.
2. With Hermes never installed, every existing flow (create/enter/rename Backpack, attach/preview/open/detach things, notes, pin, Desk, synthesis, revision, room conversation, history) works; the require-cache test proves no `app/hermes/*` module loads on those paths.
3. From a clean machine, the setup panel installs the pinned Hermes (hash-verified), starts it, completes Nous sign-in, and the echo test reports "Tested working".
4. The Assistant button is visible and functional in the world view and inside a Backpack.
5. `Ctrl+Alt+Q` toggles the companion from anywhere in Windows while Papers runs; the shortcut failing to register does not break startup.
6. A Global conversation streams a reply (fake in CI; real Hermes in manual acceptance).
7. A Backpack-scoped conversation created inside a Backpack carries that Backpack's identity in its envelope and its chip.
8. `assistant:setScope` after the first message returns the locked-scope error; no code path mutates a locked scope (test-pinned).
9. With a run streaming in a conversation scoped to Backpack A, entering Backpack B: the stream continues uninterrupted, the conversation stays selected, and its scope chip is unchanged.
10. Attaching a file resource shows the exact byte/entry share detail before send, and the envelope contains exactly that content.
11. Streamed text renders incrementally and the final text matches `message.complete`'s payload.
12. Tool events render as activity lines with humanized names; no raw payload dump appears anywhere.
13. An approval request pauses the run until [Allow this once]/[Don't allow]; the chosen `approval.respond` reaches the (fake) gateway with the right session_id.
14. A clarification round trip works with both a choice click and free text.
15. [Stop] during streaming issues `session.interrupt` and the conversation shows the honest interrupted state.
16. After quitting and reopening Papers, selecting the conversation shows its full local projection; with (fake) Hermes running, `session.resume` is issued and rebinds; with resume failing, the documented fresh-session fallback message appears.
17. Every §19 row produces its specified user-facing state and recovery action (fake-driven where automatable, manual otherwise).
18. The contract-test deep scan finds no token, port, or gateway URL in any IPC response or pushed event; the renderer cannot invoke anything outside §10.1/§10.5.
19. Events with reasoning types/fields never appear in any persisted file or any pushed event (test-pinned, mirroring P1's storage tests).
20. All pre-existing P2 tests pass unmodified.
21. All new tests pass; `npm test` runs the whole set from `app/`.
22. The packaged Windows build passes the §20.4 checklist, including: no `hermes.exe` outliving Papers, logs present, world backup diff clean.
