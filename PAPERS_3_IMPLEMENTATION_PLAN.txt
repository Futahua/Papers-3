## 1. Executive product summary

Papers 3 is the Papers 2 product — Backpacks first, one active Backpack, honest references to real things — with **Papers AI** attached: a Papers-native way to reach Hermes from anywhere in Papers. Hermes does the AI work: reasoning, planning, research, browsing, terminals, files, computer use, and delegation to coding tools such as Codex, Claude Code, and OpenCode. Papers contributes only what is uniquely Papers: the application shell, Backpacks, the active Backpack, conversation scope (`Global` or `This Backpack`), explicit resource sharing, a truthful display of Hermes' public output and requests, and the minimum lifecycle safety needed to start, reach, and stop a private Hermes reliably.

The finished Papers 3 (§29-style completion in §27) is **one coherent Papers AI experience**: the creator opens Papers, lands in their last active Backpack, reaches Papers AI from anywhere, chooses a visible scope before the first message, shares resources deliberately, watches Hermes' public activity, answers approvals and clarifications, can always Stop, and finds their old Backpack conversations migrated into the new experience. Papers 2's current Claude-CLI/Ollama path may coexist **temporarily** during implementation; the finished product removes that competing creator-facing AI surface after Hermes parity, migration, and rollback are proven (§23, Slice 8).

What Papers 3 deliberately does **not** contain: any Papers-designed AI behavior, any second agent framework, any provider catalogue, any orchestration or policy engine, any terminal or coding layer, any AI memory system competing with Hermes' sessions. Every such capability is reached through Hermes or an existing product, per the ownership boundary in §9 and the audits in §4 and §10.

The implementation proceeds in eight experience-first slices (§26), each giving the creator a visible, testable improvement, each with a rollback point, starting from a verbatim, provenance-recorded import of the verified Papers 2 tree into this repository (§2, §12).

**Source-provenance decision (Binding decision).** The verified Papers 2 tree becomes P3's starting product tree by a **verbatim tree import, not a history merge**:

1. In a P3 working clone, add P2 as a temporary remote and fetch commit `54ecce58…` only.
2. Materialize that exact tree at the P3 root (`git restore --source=54ecce581eedf059592a7ae05739911939499291 --worktree --staged -- .` after `git read-tree`-style staging, or equivalently `git archive` from the fetched commit extracted into the clean tree).
3. Verify byte-identity: `git diff --stat 54ecce58… HEAD -- .` over the imported paths must be empty before committing.
4. Add `PROVENANCE.md` at the P3 root recording: source repository, source commit `54ecce581eedf059592a7ae05739911939499291`, import date, the verification command used, and the rule that P2 remains the read-only reference for that history.
5. Commit as a single import commit whose message records the exact source commit; remove the temporary remote.

Why not merge histories: P3 already has its own root commit, and merging unrelated roots (`--allow-unrelated-histories`) buries provenance in a merge knot instead of stating it; the full, unaltered history remains permanently available in P2 itself. Why not squash-copy without fetch: fetching the real commit lets the byte-identity check run against the true source object rather than a local checkout that could have drifted. (Creator requirement: preserve source provenance and record the exact Papers 2 source commit; do not blindly merge unrelated Git histories.)

Ongoing rule: after the import, P2 and P1 are never written to; P3 is where all Papers 3 work happens.

## 5. Creator experience

The planning center. Everything else in this document exists to produce exactly this, with the least Papers-specific code. (Creator requirement, all items.)

1. The creator opens Papers.
2. Papers restores the last active Backpack when it still exists and is readable.
3. Otherwise, Papers shows the Backpack list and explains why restoration was not possible ("Your last Backpack \"<title>\" is no longer readable/was not found").
4. Backpacks remain the primary working surface — the Papers AI drawer is a collapsible companion to the Backpack, never the shell (**Observed** precedent: P2 — `app/ui/style.css:151–152` "It must never grow into the shell").
5. Papers AI is reachable from anywhere: a persistent shell-level button, the drawer, and `Ctrl+Alt+Q`.
6. Before the first message, the creator clearly sees and may choose: **Global** or **This Backpack**.
7. Scope becomes immutable by one deterministic rule (§16.2): it locks the moment Papers durably records the first message, after Send and before Hermes runs.
8. The creator asks normally. No infrastructure concepts appear (§6).
9. The creator may explicitly attach Papers resources, and sees exactly what will be shared before sending (byte counts, truncation, missing — the existing honest-preview idiom, **Observed:** P2 — `app/main.js:165–177`).
10. Papers sends only the instruction, scope identity, minimal metadata, and the selected resources (§18).
11. Hermes performs the work using its own behavior and tools. Papers adds nothing to how Hermes thinks, plans, researches, codes, browses, delegates, retries, or completes.
12. Papers displays Hermes' public response and public requests as they arrive (§19).
13. The creator may approve, clarify, or Stop at any time.
14. Switching Backpacks never cancels, retargets, or recontextualizes an existing conversation.
15. The companion accesses the same conversation and the same Hermes session — never a second AI.
16. Conversations can be reopened through their Hermes session mapping; Hermes is the authority on AI memory (§17).
17. Coding tools, research, file work, and computer use happen through Hermes (§22).
18. Failures are attributed honestly: Papers, Hermes, model/provider, external tool, permissions, network, website, or computer use (§24).
19. The creator is told whether retrying is useful, in each failure state (§24).
20. Existing Papers data remains safe and reversible throughout all transition work (§23, §28).

This plan does not describe Hermes' hidden internal workflow anywhere, and no section below prescribes one.

* New IPC namespace: `papersAI:*`; new preload namespace: `window.papersAI`; new storage folder: `<worldDir>/papers-ai/` (the folder lives beside existing data for compatibility; "worldDir" itself is a legacy code label).

## 7. Product invariants

True at every slice boundary (each is Creator requirement or Binding decision as noted):

1. Backpacks remain primary; one active Backpack at a time; Papers AI never becomes the shell. (Creator requirement)
2. The last active Backpack is restored when readable; otherwise the Backpack list with an honest explanation. (Creator requirement)
3. Existing Papers 2 data (world/Backpacks/things/notes/Desk/activity/conversations) remains readable and writable by the existing code, byte-compatible, at every slice. (Creator requirement)
4. Every existing Backpack workflow works with Hermes absent, stopped, or broken; no `app/papers-ai/*` module loads on those paths (test-pinned, mirroring **Observed:** P2 — `app/test/world.test.js:585–594`). (Binding decision)
5. Papers AI is globally reachable (button, drawer, `Ctrl+Alt+Q`). (Creator requirement)
6. Scope is chosen visibly before the first message and locks by the §16.2 rule; it never silently changes; Backpack switching never retargets a conversation. (Creator requirement)
7. Context is explicit and bounded; nothing is dumped (§18). (Creator requirement)
8. Hermes owns all AI behavior (§9); Papers never specifies it. (Creator requirement)
9. Papers displays only public, non-reasoning Hermes information; hidden reasoning is never persisted or displayed. (Creator requirement; mechanism **Observed:** P1 — `storage.rs:566–606`)
10. Hermes is authoritative for AI memory; Papers keeps only navigation/display state and never overwrites or merges Hermes history (§17). (Creator requirement)
11. Failures are visible, honestly attributed, and state whether retrying is useful (§24). (Creator requirement)
12. Secrets (Hermes token, provider credentials) and raw infrastructure never reach a renderer; Hermes stays loopback-only. (Binding decision; mechanism **Observed:** P1 — `gateway_proxy.rs`, `runtime.rs:305–330`)
13. No Hermes process outlives Papers; installation data, credentials, and runtime state are never committed. (Binding decision)
14. Coexistence with the old AI path is temporary; the finished product has one Papers AI experience (§23). (Creator requirement)
15. Every migration step is idempotent, lossless, backed up, reversible, and tested on copies of real data first (§23, §28). (Creator requirement)

* a policy engine, approval-policy system, or completion-receipt system;
* a provider manager, provider catalogue, or model catalogue;
* a tool router, tool framework, or computer-use engine;
* an AI runtime or a replacement for Hermes, Codex, Claude Code, or OpenCode;
* a second AI memory system or a Papers-owned session-history authority;
* a Papers-designed AI activity model, progress model, or phase model;
* behavioral prompts (personality, planning, research, coding, tool-selection, retry, delegation, success-criteria, workflow, computer-use, or provider instructions);
* a browsing strategy or anti-bot bypass of any kind;
* universal Backpack ontology (files/things/notes/Desk as universal capabilities);
* speculative enterprise lifecycle machinery, extensibility layers, or "future-proof" abstractions;
* new repositories, history merges, or edits to P1/P2.

## 9. Hermes / Papers ownership boundary

The binding boundary (Creator requirement, reproduced as the contract every later section must satisfy):

**Hermes owns:** AI behavior; reasoning and planning; prompts beyond the creator's instruction and explicitly supplied factual Papers context; agent sessions and resumable memory; provider and model execution; provider authentication; model discovery; research and browsing; terminal and process tools; file tools; computer use; tool selection and execution; delegation to Codex, Claude Code, OpenCode, and other supported tools; retries; approvals and clarifications as agent protocol concepts; tool errors; progress and execution events; interruption semantics; completion semantics; coding workflows; computer-use workflows.

**Papers owns only:** the Papers application and Backpacks; the active Backpack; restoring the creator's last active Backpack; a Papers-native access point for Hermes; Global or Backpack conversation scope; displaying the proposed scope before the first message; explicitly selected Papers resources; passing the creator's instruction and minimal factual context to Hermes; displaying Hermes' public non-reasoning output and public events; forwarding Hermes approval requests to the creator and returning the creator's response; forwarding clarification requests and responses; forwarding Stop; mapping Papers conversations to Hermes sessions; preserving only the minimal public display state the Papers experience needs; keeping secrets and raw infrastructure out of the renderer; stating honestly when Hermes is unavailable; the minimum lifecycle integration required to start, reach, and stop Hermes safely.

The integration model (Creator requirement):

```
Creator
  ↓
Papers interface
  ↓
Global or Backpack scope
  ↓
Explicitly selected Papers context
  ↓
Thin Papers-to-Hermes adapter
  ↓
Hermes
  ↓
Hermes sessions, models, tools, coding products,
research, browsing and computer use
```

The adapter **translates**; it never orchestrates. It introduces no competing agent model, workflow engine, session-memory system, provider system, tool framework, or coding layer. The Papers-owned list is not expanded anywhere in this plan; every proposed component in §14/§25 maps onto exactly one Papers-owned bullet, and §10 proves nothing else slipped in.

## 16. Papers AI conversation and scope model

### 16.1 Record (persisted per conversation; full persistence rules in §17)

`<worldDir>/papers-ai/conversations/<conversationId>.json` — fields: `schema:1`, `id` (`conv_…`, existing id helper — **Observed:** P2 — `app/world/ids.js:5–7`), `title` (first message, ≤70 chars, renameable), `scope` (`{kind:'global'}` or `{kind:'backpack', backpackId, titleAtLock}`), `scopeLockedAt`, `hermesSessionId|null`, `createdAt/updatedAt`, `publicStatus` (one of the §12-of-direction neutral strings plus `Idle`), `pendingApproval|null`, `pendingClarification|null`, `selectedResources[]` (references only), `displayCache[]` (§17), `lastFailure|null` (§24 shape).

New records use `backpackId` exclusively (V9); `roomId` appears nowhere in `papers-ai/` files.

### 16.2 The one deterministic scope rule (Creator requirement, V2)

**Scope locks when the first creator message is durably recorded by Papers immediately after the creator presses Send, before Hermes execution begins.**

Mechanically: `adapter.send()` first appends the creator entry (with the conversation's chosen scope) to the conversation record via an atomic write; the moment that write returns, `scopeLockedAt` is set in the same write. Only then does the adapter touch Hermes (`session.create` / `prompt.submit`). Consequences, all binding and test-pinned:

* A Hermes connection or execution failure cannot make scope ambiguous — the message and locked scope are already on disk; the entry shows an honest failure state with Continue/retry (§24 F3).
* A failed first submission never silently unlocks or retargets the conversation; retry re-uses the recorded message and scope.
* The first message is always visibly associated with its scope (the chip sits above it).
* The UI shows the lock: before lock the chip is a Global/This Backpack chooser labeled "(locks with your first message)"; after lock it is static with tooltip "Scope locked with the first message."
* Switching the active Backpack never changes any existing conversation's scope (the adapter reads scope only from the record after lock; the active Backpack feeds only *defaults for new conversations*).

### 16.3 Choosing scope

New conversations default to `This Backpack` when created while a Backpack is active, else `Global`; the chooser is visible before the first Send (Creator requirement §8.6). `setScopeBeforeLock` is refused with an honest message once `scopeLockedAt` is set — there is no unlock API.

### 16.4 Backpack changes, renames, deletion

Rename: scope stores `backpackId`; the chip shows the live title via G7. Unreadable/missing Backpack: chip shows "Backpack no longer available: \"<titleAtLock>\""; conversation stays readable and usable; resource picking via that Backpack is disabled; context carries the honest one-line fact (§18). Switching Backpacks regroups the conversation list (This Backpack / Global / Other Backpacks) and nothing else.

### 16.5 Session mapping and reopening

`hermesSessionId` binds at first successful `session.create` (`stored_session_id ?? session_id` — **Observed:** P1 — `use-agent.ts:481–496`). Reopening a conversation with a binding issues `session.resume` when connected; a changed returned id rebinds; resume failure clears the binding and states plainly that the next message starts a fresh Hermes session with the same scope (§17.3, §24 F9). Hermes' resumable memory is the continuity; Papers never fabricates it.

### 16.6 Companion rule (Creator requirement, V8)

The companion never silently creates and submits into a Backpack-scoped conversation. Its empty state offers: (a) pick an existing conversation, or (b) a visible scope chooser (Global / This Backpack "<title>") that must be tapped before its first Send is enabled. The chosen scope is displayed at the moment of submission.

## 17. Minimal persistence and Hermes authority

**Hermes is authoritative** for session history, resumable agent memory, tool state, and execution history (Creator requirement). Papers persists only what the Papers experience needs to navigate and render:

* the §16.1 record fields (ids, scope, title, timestamps, public status, pending approval/clarification, selected-resource references);
* a **redacted public display cache** (`displayCache[]`): the creator's messages, Hermes' public text, grouped public tool lines, and neutral status lines — exactly what the drawer rendered, so conversations remain readable immediately and offline. It is a cache, not memory: it is never sent back to Hermes, never merged into Hermes state, and never treated as the truth about what Hermes remembers.

**Prefer Hermes when available (Creator requirement):** on reopen with a live connection, `session.resume`'s returned `messages[]` (**Observed:** P1 — `use-agent.ts:654–681`) is rendered as the authoritative public history for the messages it covers; the display cache fills gaps (tool lines, status lines) and serves offline reads.

**Disagreement handling (Creator requirement):** if Hermes' returned history and the cache disagree materially (e.g., messages the cache has that Hermes lacks, or vice versa), Papers does not declare itself correct, does not overwrite Hermes, and does not merge speculatively. The conversation shows a visible resynchronization state: "Papers' saved view and Hermes' memory of this conversation differ. Showing Hermes' record; Papers' saved view is kept unchanged." — with an action to view the cached version. Both records persist until the creator moves on; nothing is deleted.

**Redaction before persistence and display:** every event passes the G3 redaction guard (mechanism **Observed:** P1 — `storage.rs:566–606`); reasoning event types and fields never reach `displayCache`, the renderer, or logs. (Creator requirement: never display or persist hidden reasoning.)

**Corruption:** an unreadable conversation file is listed as "unreadable conversation," never auto-deleted; delete = move to `papers-ai/trash/` (rename, same volume). Loader refuses `schema > 1` honestly.

**What Papers deliberately does not store:** raw Hermes frames, event logs, provider data, tokens, model lists, tool state, or any second copy of Hermes session internals. (Non-reinvention: no second AI memory system — audit A7.)

## 18. Explicit Papers context

**What one Send transmits to Hermes (Creator requirement; the complete list):**

1. the creator's instruction, verbatim;
2. scope identity: "Global" or the Backpack's id + title;
3. minimal Backpack metadata: the Backpack description when present, capped 500 chars (**Observed** source field: P2 — `app/world/store.js:144–158`);
4. resources explicitly selected **for that message**, read at send time through G7's truthful readers with their existing caps and honest truncation/missing markers (**Observed:** P2 — `app/world/things.js:58–118`);
5. one line of neutral factual framing identifying the block as Papers context.

Rendered deterministically (fixed order, attach-order resources) as plain text ahead of the instruction — the mechanism P1 proved for contextual prompts (**Observed:** P1 — `use-agent.ts:498` — `` `${context}\n\n${text}` ``):

```
[Papers context — supplied by the creator through Papers]
Scope: Global — the creator's Papers workspace.        |or|  Scope: the Backpack "<title>".
Backpack description: <text>                            (only when present)
Selected resource 1 of N: <name> (<kind>, <status>, at <path>)
<content as read, with the reader's own truncation/unreadable markers>
[End of Papers context]

<the creator's instruction>
```

**No behavioral super-prompt (Creator requirement, V12):** no personality, planning, research, coding, tool-selection, retry, delegation, success-criteria, workflow, computer-use, or provider instructions — the framing above contains no imperative addressed to the model, and a snapshot test enforces that (§26 Slice 5 checks).

**Explicitness (Creator requirement):** nothing is inferred or dumped — not the complete Backpack, all files, Desk contents, activity history, notes, previously attached resources, or the whole workspace. Resources are chosen per message via the picker (Backpack scope: G7's list; Global scope: the native file dialog, main-process owned — pattern **Observed:** P2 — `app/main.js:62–73`), each chip showing the exact share detail before Send (wording **Observed:** P2 — `app/main.js:165–177`). Missing resources at send time are included as their honest missing/unreadable form, exactly as previewed.

**Limits:** per-resource caps are the existing reader caps; per-message: ≤ 8 resources, ≤ 192 KiB rendered context — exceeding either is refused before sending with "That's more than one message can carry — remove something or send in parts." (Bounded, honest, no silent dropping.)

**Unavailable scope:** when a Backpack-scoped conversation's Backpack is unreadable, item 2 becomes the honest fact "Scope: a Backpack that is no longer available in Papers (\"<titleAtLock>\")." and items 3–4 are omitted.

   * **scoped per Backpack** and **tested against copies of real data** before running on live data.
4. **Default:** after successful validation, the migrated conversation becomes that Backpack's default Papers AI conversation.
5. **Removal:** the old creator-facing surfaces (the `room:say` conversation panel and the engine-backed action buttons) are removed **only after** parity, migration, and rollback are proven; the engine seam itself stays internally present until the rollback window closes (§28), then is retired. ## 28. Rollback plan

* **Slice granularity:** every slice's commits revert cleanly in reverse order; rehearse `git revert` of each slice head on a scratch branch with the suite green (executor checklist step 24).
* **Runtime rollback:** deleting `%LOCALAPPDATA%\Papers\runtime` (and `data/hermes` if desired) returns Papers to the honest unavailable state; world data is untouched by design (§20).
* **Data rollback:** the world directory is backed up before any real-data testing (checklist step 4) and before Slice 8's live migration; migration reversal = delete created conversations + migration records (originals were never modified); backups retained under `papers-ai/migrations/backup/`.
* **Transition rollback:** the Slice 8 routing and removal land as separate commits; reverting them restores the old surfaces, whose storage (`conversation.json`, engine seam) was retained through the rollback window. The window closes — and the engine path is deleted — only after the creator signs off on the unified experience in real use.
* **Scope of no-rollback:** nothing in this plan performs an irreversible operation on user data; the only destructive act anywhere is the creator manually emptying `papers-ai/trash/`.

## Appendix — Final review questions, answered

1. *Is this something the creator directly experiences?* Every slice ends in a creator-visible result (§26); §31 traces each requirement to its surface.

6. *Is Papers taking authority over Hermes session memory?* No: §17 makes Hermes authoritative, the cache a cache, and disagreement an honest visible state.
7. *Present requirement or speculative abstraction?* Each glue module's build-vs-reuse record (§14) names its present §9 responsibility; §25.4 lists what was refused.
8. *Is the proposed custom code uniquely Papers-specific?* Yes — scope, Backpack compatibility, the access surfaces, and lifecycle safety; each with a removal path if upstream absorbs it (§14).
9. *Is the license verified?* Verified-at-pin steps are in the checklist; two are explicitly **License unresolved** and gate their slices (§11).
10. *Does the plan preserve existing user data?* Yes — byte-compatible import, read-only world access from Papers AI code, non-destructive migration, backups (§17, §23, §28).
11. *Is the change reversible?* Every slice, the runtime, and the migration have rehearsed rollbacks (§28).
12. *Is the scope visible and deterministic?* One rule, §16.2, visible before and after lock, tested including failure cases.
13. *Does the creator understand what is happening?* §6 vocabulary, §19 projection, §24 attribution with retry guidance.
14. *Is any failure becoming a black box?* §24 forbids bare done/failed/blocked/stopped; every row states what is known, unknown, and useful to retry.
15. *Can the same result be achieved with less Papers-specific code?* The consolidation pass (A22) got the glue to ≈1,400 lines across seven modules; every further cut identified would delete a creator-required behavior (§14).
16. *Is temporary coexistence being described as the finished product?* No — §27 separates early usable slices from finished Papers 3, and finish requires unification, migration, and removal (V4/V5).
